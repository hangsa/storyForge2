# StoryForge v1.1 设计评审报告

> 评审日期：2026-06-09
> 评审目标：对照用户四项核心需求，检查 v1.1 设计的覆盖度和缺口

---

## 评审结论总览

v1.1 在**系统能力**层面（创意发散工具、叙事资产追踪、一致性保障、记忆系统）设计扎实，但在**用户协作模式**层面存在结构性缺口。当前设计将用户定位为"审核者+异常介入者"而非"创作协作者"，这与用户明确的"各阶段都要能与 agent 充分沟通讨论"的需求存在根本性偏差。

---

## 一、逐项需求评审

### 需求一：创意构思阶段——发散、创新、可讨论

**现有覆盖：**

| 覆盖点 | 状态 | 说明 |
|---|---|---|
| 套路变异器（4 种操作） | ✅ 覆盖 | 逆转/融合/升维/打破，可生成大量变体 |
| 矛盾设定生成器（5 模板） | ✅ 覆盖 | 结构化生成强张力设定 |
| WhatIf 递归发散树 | ✅ 覆盖 | 最多 84 节点，保证发散广度 |
| 体裁融合器 | ✅ 覆盖 | 跨体裁结构层面嫁接 |
| 新颖度评估 | ✅ 覆盖 | 4 维度量化评分 |
| 故事基因（Story DNA） | ✅ 覆盖 | 结构化产出物 |
| 世界观构建 Agent | ✅ 覆盖 | 单独阶段、结构化产出 |
| 角色设计 Agent | ✅ 覆盖 | 完整人物档案 |

**缺口：**

| 缺口 | 严重度 | 问题描述 |
|---|---|---|
| **无迭代讨论机制** | 🔴 严重 | 当前模式是"用户输入一句话 → 系统输出 Story DNA → 用户审核"。用户只能通过或拒绝，无法与 Creative Director 进行多轮对话式的创意碰撞。"充分沟通讨论"完全未体现。 |
| **创意阶段无分支探索** | 🔴 严重 | CreativeOS 的 7 个引擎生成结果后，用户看到的是"最终优选结果"。但创意讨论的本质是：用户应该能看到发散树的全貌、选择感兴趣的分支深入、要求某个方向重新发散。当前设计缺少创意树的交互式探索界面。 |
| **世界观/角色设计无讨论环** | 🟡 中等 | Worldbuilder 和 Character Designer 在概念审核后才执行，且各自独立输出。用户无法在其中任一阶段说"这个势力设定不够好，我们讨论一下替代方案"，无法回退到上游重新调整。 |
| **"审核门"语义错误** | 🔴 严重 | `prompt` / `auto` / `skip` 三种模式都暗示审批流程，而非协作讨论。缺少 `discuss` 模式——在此模式下 Agent 主动提问、提供备选方案、邀请用户偏好表达。 |
| **创意约束只能事后指定** | 🟡 中等 | 用户只能在 Expert Config 中预先声明约束（如"禁止虐主"），但创意讨论中用户可能产生新的约束想法（"我突然觉得主角的能力应该有一个情感代价"），当前没有将讨论中产生的新约束注入系统的方式。 |

### 需求二：剧情情节设计阶段——反转、抓人、可讨论

**现有覆盖：**

| 覆盖点 | 状态 | 说明 |
|---|---|---|
| 七类叙事资产登记 | ✅ 覆盖 | 冲突/承诺/谜团/反转/揭晓/目标/期待 |
| 资产关联网络 + 级联 | ✅ 覆盖 | 自动追踪因果关系和连锁反应 |
| 全局伏笔映射表 | ✅ 覆盖 | 植物→回收生命周期管理 |
| 张力曲线 + 智能预警 | ✅ 覆盖 | 实时计算、自动检测节奏问题 |
| 大纲规划 Agent | ✅ 覆盖 | 分卷分章细纲 |

**缺口：**

| 缺口 | 严重度 | 问题描述 |
|---|---|---|
| **大纲规划无讨论环** | 🔴 严重 | 同样是"系统输出 → 用户审核"的线性模式。用户无法对大纲说"第七卷的反转设计不够惊艳，帮我重新构思 3 个替代方案"或"第 45-50 章节奏太平，在中间插入一个意外事件"。 |
| **缺少"反转头脑风暴"模式** | 🔴 严重 | 当前设计假定 Outliner 自己就能规划好反转。但最精彩的反转往往来自人与 AI 的碰撞——用户提出一个模糊的"想让某角色叛变"想法，Agent 应该能基于当前叙事资产状态展开多种叛变方案（谁叛变、什么时候、影响什么），让用户选择和发展。 |
| **伏笔密度不可讨论** | 🟡 中等 | 系统计算张力曲线，但没有让用户参与"我要在多少章内布多少个伏笔"的决策。这应该是用户与 Agent 共同讨论决定的创作策略，而非系统自动计算的黑盒。 |
| **大纲缺乏"如果...会怎样"的剧情模拟** | 🟡 中等 | 用户在审核大纲时，最大的需求是"如果我改变这个情节会怎样"。当前设计没有提供剧情分支模拟——用户无法对 Outliner 说"如果把第 30 章的反派身份揭示提前到第 20 章，会对后续剧情产生什么连锁影响？" |

### 需求三：内容创作阶段——文笔优质、一致性、适当创新、可讨论

**现有覆盖：**

| 覆盖点 | 状态 | 说明 |
|---|---|---|
| Scene Engine 叙事规格 | ✅ 覆盖 | 每幕有明确的叙事任务和读者体验目标 |
| Style Engine 三层控制 | ✅ 覆盖 | 体裁模板 + 写作公式 + 禁忌约束 |
| Reviewer 三层审稿 | ✅ 覆盖 | 事实核查 + 叙事建议 + 风格标签 |
| SF_LOG 变化标记 | ✅ 覆盖 | 确定性追踪叙事变化 |
| 连贯性评分 | ✅ 覆盖 | 5 维度加权评分 |

**缺口：**

| 缺口 | 严重度 | 问题描述 |
|---|---|---|
| **写作阶段无用户介入点** | 🔴 严重 | 整个 CHAPTER_LOOP 是全自动的——Writer 写、Reviewer 审、通过或重写。用户完全被排除在写作过程之外。用户无法在写作过程中说"这段写得好，保持这种感觉"或"这个对话太僵硬，让我自己来改"。 |
| **风格引擎抑制创新** | 🔴 严重 | Style Engine 的设计哲学是"约束"——L1/L2/L3 层层规则确保风格一致。但用户要求的是"在保持一致性的情况下能保持适当创新写法"。当前没有"创新豁免"机制——Writer 无法在某个 Scene 主动请求"此处需要突破风格规则以制造特殊效果"，并让用户批准。 |
| **"适当创新"无系统支撑** | 🟡 中等 | 系统可以检测"风格一致性"，但没有检测"风格创新性"的维度。例如：在一段标准爽文风格中突然插入一段诗意的意识流描写——这对读者可能是惊喜也可能是违和。系统应该能标记这种"风格偏离"并询问用户意图，而非简单地判定违规。 |
| **写作过程无实时讨论** | 🟡 中等 | 在 Scene 级别，Writer 规划好 3-6 个 Scene 后直接开始写作。用户可能想对某个 Scene 说"这个 Scene 我想尝试第一人称视角"——当前没有 Scene 级别的讨论机制。 |
| **用户改写无系统回馈** | 🟡 中等 | 如果用户手动修改了某个 Scene 的文本，系统只会通过 Reviewer 的一致性检查来验证。但用户改写的文本中隐含的叙事变化（新伏笔、关系变化）如果没有 SF_LOG 标记，就会被系统忽略。缺少"用户编辑后自动建议添加变化标记"的辅助机制。 |

### 需求四：角色成长曲线——一致性 + 合理成长

**现有覆盖：**

| 覆盖点 | 状态 | 说明 |
|---|---|---|
| 能力进化阶段追踪 | ✅ 覆盖 | 一阶→二阶→三阶，记录上限和代价 |
| 角色声音签名 | ✅ 覆盖 | 说话风格、思维模式、绝对禁忌 |
| 信念变化严格门槛 | ✅ 覆盖 | 多事件触发、白名单验证 |
| 核心人格层保护 | ✅ 覆盖 | 信念/欲望/恐惧/价值观不轻变 |

**缺口：**

| 缺口 | 严重度 | 问题描述 |
|---|---|---|
| **成长曲线无设计阶段** | 🔴 严重 | 当前角色设计阶段产出的是角色的"初始状态"+"进化阶段列表"。但"成长曲线"应该是被精心设计的——什么时候成长、成长多快、成长是否有代价、成长如何与剧情高潮同步。当前没有"成长曲线设计师"角色或阶段。 |
| **成长与剧情割裂** | 🟡 中等 | 角色的能力进化阶段只记录了"什么时候到什么阶段"，但没有与 StoryOS 的叙事资产关联。角色能力的重大升级应该绑定到特定的剧情转折点（反转揭示、重大冲突解决）。当前这些是独立追踪的。 |
| **成长曲线没有用户讨论** | 🔴 严重 | 用户无法与 Character Designer 讨论"主角的成长是不是太快了？第 30 章就到三阶后面没得写了怎么办？"或"我想让主角在第 50 章经历一次能力倒退作为转折"。成长曲线设计应该是用户与 Agent 深度讨论的主题。 |
| **缺少成长代价追踪** | 🟡 中等 | 好的角色成长往往有代价（获得力量→失去人性；变得更强大→变得更孤独）。当前只追踪"能力上限和代价"，但没有追踪"成长代价对角色心理和人际关系的影响"。 |

## 二、跨阶段系统性问题

### 2.1 用户协作模式缺失（最严重问题）

当前整个系统的工作流是：

```
用户 → [一次性输入] → 系统自动流水线 → [审核门] → [审核门] → 导出
```

用户期望的工作流应该是：

```
用户 ←→ [持续讨论] ←→ Agent
  ├── 创意阶段：多轮发散、比较方案、逐步收敛
  ├── 情节阶段：反复推敲、模拟推演、调整伏笔
  ├── 写作阶段：逐幕审阅、风格指导、灵感注入
  └── 修订阶段：诊断讨论、修复策略协商
```

**根本原因：** v1.1 继承了 implementation 文档的工程化设计基因——将创作流程建模为"确定性的流水线"，而创作本质上是一个"探索性、迭代性、对话性"的过程。

### 2.2 需要在设计中补充的核心概念

| 概念 | 说明 | 适用阶段 |
|---|---|---|
| **对话环（Dialogue Loop）** | 每个阶段内，Agent 主动提问、提供备选、邀请反馈，支持多轮来回。用户不完全满意时可以要求重新生成或指定方向。 | 全部阶段 |
| **创意画布（Creative Canvas）** | 一个可视化的创意探索空间，展示发散树的全貌，用户可以点击节点深入、标记偏好、剪枝。不是"输入→输出"的黑盒，而是"探索→选择→深入"的交互。 | 创意构思、情节设计 |
| **分支模拟（Branch Simulation）** | 允许用户在任意决策点说"如果我把这个设定改了会怎样"，系统基于叙事资产网络推演连锁影响，呈现前后对比。 | 情节设计、大纲规划 |
| **风格沙盒（Style Sandbox）** | 在正式写作前，用一小段文本测试不同风格参数的效果，用户选择最满意的风格配置。 | 写作准备 |
| **创新豁免（Innovation Pass）** | Writer 可以在某个 Scene 中主动标记"此处有意突破风格规则"，附带创新意图说明，提交用户审批。用户批准后该 Scene 不受风格约束审查。 | 内容写作 |
| **成长工坊（Growth Workshop）** | 角色设计阶段的一个子模式，用户与 Agent 共同设计角色的完整成长弧线：起点→关键转折点→代价→终点。产出物是"成长曲线图"而非简单的阶段列表。 | 角色设计 |
| **章节评审会（Chapter Review）** | 每章写完后不仅是 Reviewer 自动审稿，还有一个用户与 Agent 的评审对话——用户可以对任何 Scene 提出修改意见、讨论替代写法、标记"这段灵感保留到后续使用"。 | 内容写作 |

## 三、v1.2 改进建议优先级

### P0（必须修改——缺少则核心需求不满足）

1. **将"审核门"升级为"协作门"（Collaboration Gate）**
   - 在 CONCEPT_GEN 后增加对话环：Creative Director 展示 3-5 个 Story DNA 变体供用户比较选择，或用户不满意则指定方向重新生成
   - 在 OUTLINE_PLANNING 后增加对话环：Outliner 展示大纲后，主动提问"哪几卷的反转需要加强？"、"有没有想让某个角色提前/延后出场？"
   - 在每章写作前增加"Scene 规划评审"：用户可以调整 Scene 顺序、增删 Scene、指定某 Scene 的风格实验
   - 在每个阶段增加 `discuss` 门控模式（区别于 `prompt` 的审批模式）

2. **在创意阶段增加"创意画布"交互模式**
   - WhatIf 树的交互式探索界面（而非仅输出叶子节点）
   - 用户可以点击树节点，要求以该节点为新的根节点重新发散
   - 用户可以标记"喜欢"的创意分支，系统基于偏好重新加权排序

3. **在情节设计阶段增加"分支模拟"**
   - 用户对大纲的任意修改请求 → 系统推演连锁影响 → 展示对比
   - "如果删除第 30 章的反转会怎样？" → 张力曲线变化、受影响伏笔列表、替代建议

### P1（应该修改——显著提升协作体验）

4. **在写作阶段增加 Scene 级别的用户介入点**
   - 每个 Scene 写完后展示给用户（而非等到整章完成）
   - 用户可以选择：通过 / 要求重写 / 自己修改 / 标记风格偏好
   - 用户手动修改后，系统自动建议需要添加的 SF_LOG 标记

5. **增加风格沙盒和"创新豁免"机制**
   - 正式写作前用小段文本测试不同风格参数
   - Writer 在写作时可以主动申请创新豁免（"此处有意突破常规"）
   - 突破的规则、原因、效果被记录，用于后续风格提炼

6. **在角色设计阶段增加"成长工坊"**
   - 角色设计不只是填写档案，而是用户与 Agent 共同讨论成长弧线
   - 成长阶段与剧情里程碑绑定（"第 X 次反转揭示后 → 能力升级 + 心理代价"）
   - 追踪成长代价对角色关系和人格的影响

### P2（建议修改——锦上添花）

7. **每章完成后增加"章节评审会"**
   - Reviewer 自动审稿后，用户与 Agent 进行简短的评审对话
   - 讨论本章的亮点、不足、可以带入下一章的灵感
   - 用户标记"这段写法我特别喜欢，后续保持"

8. **增加"灵感捕捉"机制**
   - 在任何阶段的讨论中，用户和 Agent 都可能产生新的灵感（不限于当前阶段）
   - 这些灵感被自动捕捉到 Idea Pool，供后续阶段调用
   - 避免"讨论中产生的好想法被遗忘"

## 四、总结

v1.1 是一份优秀的**系统能力设计文档**——它清晰定义了 StoryForge 能做什么。但它不是一份完整的**产品设计文档**——因为它没有定义用户如何使用这个系统进行创作协作。

核心问题可以归结为一句话：**当前设计将用户建模为"流水线的检查点"，而用户需要的是"坐在身边的创作伙伴"。**

修复方向明确：在每个阶段内部增加**对话环**机制，将线性审批变为多轮协作，将黑盒输出变为可探索的创意空间。这些修复不改变底层系统能力的架构（CreativeOS、StoryOS、MemoryOS 等仍然成立），但需要在上层交互模式上做根本性的重新设计。
