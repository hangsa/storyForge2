# StoryForge 1.4 — 产品设计文档

> v1.4 变更：基于 v1.3 审查报告修复。修正信号仲裁表中硬编码阈值与体裁差异化阈值的冲突、新增 Mid-Scene 崩溃恢复机制、STAGE 5 增加重构区域专项诊断、补充语义预检开销估算、澄清 Global Trope Pool 数据更新方式为非实时爬取而是随版本快照、统一矛盾模板存储层级描述、修正 Fact Guard "零 LLM 调用"表述精度、恢复 STAGE 2/3 完整数据结构。所有系统模块的描述均为完整独立内容，不依赖之前版本。

## 核心理念

StoryForge 是一个**人机协作**的创意叙事操作系统。它不是替代作者，而是成为作者的创作伙伴——在创意构思时可以头脑风暴，在情节设计时可以推演对比，在内容写作时可以逐幕审阅和注入灵感。

三个核心目标：

| 核心目标 | 能力链 | 负责系统 |
|---|---|---|
| 构思足够发散、足够创新 | Creative Chain | CreativeOS + Creative Director |
| 剧情足够反转、足够抓人 | Narrative Chain | StoryOS + ReaderOS + Scene Engine + Style Engine |
| 角色/剧情高度一致 | Consistency Chain | MemoryOS + State Machines |

**设计哲学：** 系统负责骨架（一致性、节奏、读者体验），AI 负责血肉（创意发散、文字写作），**用户全程参与决策和灵感注入**。不是"输入→自动输出"的黑盒，而是"讨论→探索→选择→深入"的协作过程。

## 二、Conductor 主控编排器详设

> v1.3 新增。v1.2 中 Conductor 仅有架构图中的职责标注，缺乏具体设计。本节给出完整的阶段状态机、回退影响传播、信号仲裁和熔断降级规格。

### 2.1 阶段状态机

Conductor 维护一个六阶段有限状态机，控制创作流程的前进、回退和跳转。

```
                    ┌──────────────────────────────────────┐
                    │           任意阶段均可回退到          │
                    │           之前的任意阶段              │
                    └──────────────────────────────────────┘
                                      │
INIT ──→ STAGE1 ──→ STAGE2 ──→ STAGE3 ──→ STAGE4 ──→ STAGE5 ──→ STAGE6
                                      │
                    ┌──────────────────────────────────────┐
                    │         STAGE 4 内部可循环            │
                    │   (逐章写作，章间可回退到 STAGE 2-3)   │
                    └──────────────────────────────────────┘
```

**状态转换规则：**

| 转换 | 触发条件 | 前置校验 |
|---|---|---|
| INIT → STAGE1 | project.json 创建完成 | 必选字段完整性校验 |
| STAGE1 → STAGE2 | concept.json + story_dna.json 产出 | 新颖度评分 ≥ 40（低于则警告但允许强行推进） |
| STAGE2 → STAGE3 | world.json + characters.json 产出 | 核心角色 ≥ 1，世界观核心字段完整 |
| STAGE3 → STAGE4 | outline.json 产出（含伏笔映射 + 成长绑定） | 每卷至少 1 章，全局伏笔映射表非空 |
| STAGE4 → STAGE5 | 全部章节写作完成 | 章节数 = 大纲总章数，每章连贯性评分记录完整 |
| STAGE5 → STAGE6 | diagnosis_report.json 中 P0 问题已处理（修复或用户确认跳过） | 无未处理的 P0 问题 |
| **任意阶段 → 之前阶段** | 用户主动发起回退 | 执行回退影响传播分析（见 2.2 节） |

### 2.2 回退影响传播机制

当用户在第 N 阶段决定回退修改第 M 阶段（M < N）的产出物时，Conductor 执行以下流程：

**Step 1 — 影响范围计算（确定性）：**

Conductor 遍历所有已产出数据，识别受回退影响的元素：

```
输入：回退目标阶段 M，修改的产出物列表
输出：影响范围报告

计算逻辑：
1. 如果修改了 Story DNA（STAGE1）：
   - 标记所有基于旧 Story DNA 的 world.json、characters.json、outline.json 为"待复核"
   - 标记所有已完成章节为"待复核"（因为核心设定可能改变）

2. 如果修改了 world.json / characters.json（STAGE2）：
   - 扫描 outline.json 中引用了被修改元素的伏笔、成长绑定
   - 扫描已完成章节的 SF_LOG 中涉及被修改角色/地点/规则的标记
   - 列出受影响的章号清单

3. 如果修改了 outline.json（STAGE3）：
   - 对比新旧大纲的章节结构差异
   - 标记结构发生变化的章节为"需重写"
   - 标记伏笔映射表中受影响的伏笔

4. 如果仅修改 style_formula.yaml：
   - 影响范围限于后续写作，已完成章节不强制重写
```

**Step 2 — 影响报告展示：**

系统向用户展示分级影响报告：

```
┌─────────────────────────────────────────┐
│         回退影响分析报告                  │
│                                          │
│  回退操作：STAGE4 → STAGE2               │
│  修改对象：characters.json（林峰成长曲线） │
│                                          │
│  P0 必须重写（1 章）：                    │
│    第 26 章 — 成长转折点 tp_001 章节绑定   │
│    变更，原 SF_LOG 与新版成长曲线冲突       │
│                                          │
│  P1 建议复核（3 章）：                    │
│    第 27-29 章 — 角色能力阶段描述可能      │
│    与新版能力阶段设定不一致                │
│                                          │
│  P2 无影响（96 章）：                     │
│    其余章节不受此次修改影响                │
│                                          │
│  叙事资产影响：                           │
│    冲突 cf_003 关联的成长事件章节变更      │
│    → 需更新冲突的 expected_resolution     │
│                                          │
│  [确认并执行]  [取消回退]  [仅更新元数据]  │
└─────────────────────────────────────────┘
```

**Step 3 — 用户决策与执行：**

- **确认并执行：** 系统更新目标产出物，标记受影响章节为 `status: pending_rewrite`，在 STAGE 4 恢复时优先处理
- **取消回退：** 保留现有数据不变
- **仅更新元数据：** 修改产出物但标记已有章节为"兼容模式"——不强制重写，但在连贯性评分中增加一条"与当前设定存在已知偏差"的标注

**Step 4 — SF_LOG 对齐（如涉及已完成章节）：**

对于标记为"必须重写"的章节，Writer 在重写时自动获取新版设定作为上下文。章节重写完成后，StoryOS Agent 重新扫描 SF_LOG 并更新叙事资产注册表。对于"建议复核"的章节，Reviewer 在下一次章节评审会中额外检查与新版设定的一致性。

### 2.3 信号仲裁

Conductor 协调各 OS 之间的信号传递，按优先级排队：

| 优先级 | 信号类型 | 来源 | 处理方式 |
|---|---|---|---|
| P0 | 熔断触发（3 次重试失败） | Reviewer Fact Guard | 立即中断当前 Scene，通知用户 |
| P1 | 严重预警（追更欲 < 体裁对应严重阈值，疲劳度 > 体裁对应疲劳阈值） | ReaderOS | 下一 Scene 开始前注入提示 |
| P2 | 叙事资产逾期（承诺/伏笔超期） | StoryOS | 下一章 Scene 规划时提醒 |
| P3 | 状态漂移警告 | Reviewer Narrative Guard | 章节评审会中展示 |
| P4 | 风格偏离通知 | Reviewer Style Guard | 记录到 review_report |
| P5 | 灵感捕捉通知 | 灵感路由器 | 阶段切换时批量展示 |

### 2.4 熔断降级

当同一 Scene 的 Fact Guard 重试达到 3 次仍未通过：

1. **立即熔断：** 停止该 Scene 的自动重试循环
2. **降级处理：** 标记该 Scene 的问题列表，生成兼容性说明（"第 3 次重写后 Fact Guard 检查项 X 仍未通过，原因：…"）
3. **用户通知：** 强制切入 `review` 模式，展示 Scene 内容和问题说明，由用户决定：
   - 接受当前版本（标记为 `force_passed`）
   - 手动修改（进入用户编辑模式）
   - 跳过此 Scene（标记为 `skipped`，在章节组装时用占位符替代）
4. **记录归档：** 熔断事件记录到 progress.json 的 `circuit_breaker_triggers`，供 STAGE 5 诊断时重点关注

### 2.5 断点续写

Scene 级别粒度。每个 Scene 完成后写入 `.storyforge_checkpoint.json`（覆盖模式）：

```json
{
  "project_id": "proj_001",
  "checkpoint_time": "2026-06-10T14:30:00Z",
  "pipeline_stage": "STAGE4",
  "current_chapter": 42,
  "current_scene": 3,
  "snapshots": {
    "l0_runtime": { /* 当前章即时状态 */ },
    "storyos_registries": { /* 七项注册表当前状态 */ },
    "character_states": { /* 所有角色当前状态 */ },
    "readeros_state": { /* 七项读者指标当前值 */ }
  },
  "pending_alerts": ["追更欲连续3章下降"],
  "recovery_instructions": "从第42章第4幕开始，Writer 加载 L0 快照恢复上下文"
}
```

恢复时，Conductor 读取 `pipeline_stage` 和 `current_chapter`/`current_scene`，从断点继续执行。

### 2.6 Mid-Scene 草稿缓存

> v1.4 新增。断点续写以 Scene 为粒度，但 Scene 写作过程中也可能发生崩溃。Mid-Scene 草稿缓存提供更细粒度的崩溃恢复。

**机制：**

Writer 在生成每幕文本后，立即将当前 Scene 的已生成内容写入临时草稿文件 `.storyforge_draft_cache.json`（与 checkpoint 分离，避免覆盖已通过 Scene 的正式快照）：

```json
{
  "project_id": "proj_001",
  "chapter": 42,
  "scene": 3,
  "draft_content": "林峰睁开眼睛的瞬间，以为自己还在实验室里...（已生成的全部文本）",
  "completed_beats": ["setup", "mini_payoff"],
  "current_beat": "cliffhanger",
  "embedded_logs": ["character_emotion:2", "registry_create:1"],
  "last_update": "2026-06-10T14:32:15Z"
}
```

**恢复流程：**

Conductor 启动时检查草稿缓存：
1. 如果 `.storyforge_draft_cache.json` 存在且 `last_update` 晚于 `.storyforge_checkpoint.json` 的 `checkpoint_time` → 存在未完成的 Scene
2. 提示用户："检测到第 42 章第 3 幕未完成的写作草稿（上次保存：14:32）。是否从草稿继续？"
3. 用户选择：
   - **从草稿继续：** Writer 加载草稿内容作为起点，继续完成剩余 beat
   - **放弃草稿：** 从该 Scene 开头重新开始
4. Scene 完成并通过 Fact Guard 后，草稿缓存自动清除

草稿缓存文件不参与 StoryOS 解析——它只是恢复工具，不是叙事数据源。

## 三、核心协作模式

### 3.1 协作门（Collaboration Gate）

系统在每个创作阶段提供五种协作模式，用户可根据需要选择。默认模式倾向于协作而非自动化。

| 模式 | 行为 | 适用场景 |
|---|---|---|
| **`discuss`** | Agent 展示成果，主动提出开放问题，提供 3-5 个备选方向供用户比较选择。用户可进行多轮对话、指定新方向要求重新生成、标记偏好分支。 | 创意构思、情节设计（默认推荐） |
| **`review`** | Agent 展示成果，用户逐项提出修改意见，Agent 针对性调整并展示调整后的变化。修改记录可追溯、可回滚。 | 大纲细化、角色调整 |
| **`approve`** | 快速确认模式，用户只需通过或拒绝。但拒绝后自动切入 `discuss` 模式而非终止。 | 作者对方向已有明确想法时 |
| **`live`** | 用户逐幕实时跟进写作过程，每幕完成后立即展示，用户可当场调整方向、重写或标记偏好。 | 内容写作阶段（默认推荐） |
| **`auto`** | 全自动执行，仅在连贯性评分低于阈值或熔断触发时通知用户。 | 用户充分信任系统的批量写作场景 |

**关键原则：每个阶段默认开启协作模式，用户可以选择简化，但不会被动地被排除在创作过程之外。**

### 3.2 协作疲劳感知

> v1.3 新增。针对审查指出的 live 模式交互量过大风险，增加疲劳感知机制。

系统追踪用户的交互模式，在检测到以下信号时主动调整行为：

| 信号 | 判定条件 | 系统响应 |
|---|---|---|
| **快速连续跳过** | 用户连续 ≥ 10 个 Scene 选择"快速通过" | 在下一章开始前提示："你已连续快速通过 10 个 Scene。是否需要切换到 `auto` 模式？你可以在任何时候切回 `live`。" |
| **长时间写作会话** | 单次写作会话超过 90 分钟 | 在 Scene 完成后提示："当前写作会话已持续 90 分钟，建议休息。当前进度已自动保存。" |
| **深夜模式** | 当地时间 23:00-06:00 且连续操作 ≥ 30 分钟 | 温和提示："夜深了，当前所有内容已保存。休息好再写，效果更好。" |
| **决策质量下降** | 连续 ≥ 5 个 Scene 在快速通过后触发了连贯性评分下降（降幅 > 5 分） | 提示："最近几章的连贯性评分有所下降，建议切换到 `review` 模式仔细检查。" |

这些感知机制的核心理念是**温和提醒而非强制干预**——系统不会替用户做决定，只是在可能影响创作质量的时刻给出建议。所有疲劳感知均可由用户在配置中关闭。

### 3.3 灵感路由（Inspiration Router）

在任何阶段的任何讨论中产生的灵感，系统自动识别、分类、存储，确保不会遗失。

```
任何阶段的讨论内容
        │
        ▼
灵感路由器（自动运行，对用户透明）
  ├── 识别灵感信号："如果..."、"或者..."、"能不能..."、"我突然想到..."
  ├── 自动分类（含多分类冲突处理，见 3.4 节）：
  │     ├── 新设定灵感 → Idea Pool（标记来源、上下文、关联故事基因）
  │     ├── 新剧情想法 → StoryOS 暂存区（标记关联章节和角色）
  │     ├── 新角色灵感 → 角色设计建议（标记关联故事基因）
  │     ├── 风格偏好   → Style Engine 偏好记录
  │     └── 写作灵感   → Scene 灵感池（在后续写作中供 Writer 参考）
  │
  └── 阶段切换时主动提醒：
        "在概念讨论阶段你产生了3个灵感，是否要在世界观构建中参考？"
```

用户在任何阶段都可以打开"灵感面板"，查看所有历史阶段积累的灵感，拖拽应用到当前阶段。灵感面板中的每一条都记录了产生阶段、原始上下文、关联的故事元素。

### 3.4 灵感多分类冲突处理

> v1.3 新增。灵感可能同时命中多个分类，需要明确的冲突处理逻辑。

**多分类命中时的处理规则：**

| 场景 | 处理方式 |
|---|---|
| 命中 1 个分类（高置信度 ≥ 0.8） | 直接归类，不通知用户 |
| 命中 1 个分类（中置信度 0.5-0.8） | 归类并附加"待确认"标签，用户可在灵感面板中修改 |
| 命中 ≥ 2 个分类（最高置信度差距 > 0.3） | 归类到最高置信度类别，附加备选分类标签 |
| 命中 ≥ 2 个分类（置信度接近，差距 ≤ 0.3） | **同时归入多个类别**（多标签），在灵感面板中以"跨类"标记展示，用户可手动拆分或确认多分类 |

**用户纠错能力：**

灵感面板中每条灵感支持：
- **重新分类：** 拖拽到目标分类区域
- **拆分：** 如果一条灵感包含多个独立想法，拆分为多条
- **合并：** 将多条相关灵感合并为一条复合灵感
- **标记无效：** 标记为"噪声"（误识别），系统记录该模式以减少未来误分类
- **编辑内容：** 修改灵感的描述文本和关联元素

**分类置信度计算：**

灵感路由器使用轻量级分类器（基于关键词 + 上下文语义特征，Tier 3 模型）对每条灵感打分。置信度在灵感面板中展示，低置信度（< 0.5）的灵感以半透明样式显示，提示用户人工确认。每一条灵感记录其分类置信度，用户修改分类后，系统记录纠错日志，用于优化分类模型。

用户的纠错行为被记录为训练数据，定期用于优化分类规则。纠错数据存储在用户本地，不上传。

## 四、INIT 阶段：项目初始化

> v1.3 新增。v1.2 中 INIT 阶段仅有输入输出标注而无任何规格描述，这是新用户接触系统的第一个界面。

### 4.1 设计目标

INIT 阶段完成两件事：收集创作意图的最小必要信息以生成 `project.json`，以及引导用户完成风格沙盒的初始配置。整个过程设计为**向导式对话**，约 5-10 分钟，可以从零开始也可以从已有想法开始。

### 4.2 初始化向导流程

**Step 1 — 意图输入：**

系统展示一个开放式输入界面：

```
┌─────────────────────────────────────────┐
│        欢迎使用 StoryForge               │
│                                          │
│  你想写一个什么样的故事？                 │
│                                          │
│  可以是一句话：                           │
│  "顶级程序员重生回到大学，发现世界的        │
│   底层是一套可以修改的源代码"              │
│                                          │
│  可以是一段话：                           │
│  （描述故事的核心设定、主角、世界观...）    │
│                                          │
│ 也可以上传已有章节或大纲                   │
│  [上传文件]                               │
│                                          │
│  不知道写什么？让系统帮你发想              │
│  [给我一些热门方向的灵感]                  │
│                                          │
│  [开始]  [跳过，以后再说]                 │
└─────────────────────────────────────────┘
```

如果用户选择"给我一些热门方向的灵感"，系统从 Global Trope Pool（见第七节）中提取当前市场蓝海方向，展示 5-8 个热门创意种子及其市场饱和度，用户可选择感兴趣的方向作为起点。

**Step 2 — 基础约束设定：**

用户设定创作的基础参数：

```
┌─────────────────────────────────────────┐
│         基础创作参数                      │
│                                          │
│  目标总章数：    [100] 章                 │
│  最低单章字数：  [4000] 字                │
│  目标读者定位：  [男频 ▼]                 │
│                  ├─ 男频                 │
│                  ├─ 女频                 │
│                  └─ 不限                 │
│  创作语言：      [简体中文 ▼]             │
│  预期更新频率：  [日更]                   │
│                                          │
│   [确认]  [使用推荐默认值]                │
└─────────────────────────────────────────┘
```

**Step 3 — 风格方向初选：**

系统从内置体裁模板中展示可选方向，用户可预览每种体裁的典型文本风格：

```
┌─────────────────────────────────────────┐
│         选择目标体裁                      │
│                                          │
│  ○ 爽文（Cool Novel）                    │
│    短句快节奏，高频爽点，强钩子结尾         │
│    [预览示例段落]                         │
│                                          │
│  ○ 严肃文学（Literary）                   │
│    长句铺陈，心理描写深度优先，节奏舒缓     │
│    [预览示例段落]                         │
│                                          │
│  ○ 悬疑推理（Mystery）                    │
│    线索驱动，信息差维持，逻辑自洽优先       │
│    [预览示例段落]                         │
│                                          │
│  ...（更多体裁）                          │
│                                          │
│  ☐ 上传参考文本（可选，用于风格提炼）       │
│                                          │
│  [下一步]                                │
└─────────────────────────────────────────┘
```

**Step 4 — 确认与生成：**

系统汇总用户的输入，生成 `project.json` 和初始 `style_formula.yaml`，并展示摘要：

```
┌─────────────────────────────────────────┐
│         项目初始化完成                    │
│                                          │
│  项目名称：代码天才重生记                  │
│  目标体裁：爽文                           │
│  目标章数：100 章 × 4000 字               │
│  目标读者：男频                           │
│  风格配置：cool_novel 默认模板             │
│                                          │
│  下一步：进入概念讨论阶段                  │
│  [进入 STAGE 1]  [先去风格沙盒微调参数]    │
└─────────────────────────────────────────┘
```

### 5.2 STAGE 1: 概念讨论 【默认模式：discuss】

**输入：**

| 输入项 | 来源 | 必选 | 说明 |
|---|---|---|---|
| 用户意图 | 用户直接输入 | 是 | 一句话到一段话的故事想法，可模糊可详细 |
| 风格配置 | Style Sandbox 产出 | 是 | 用户在风格沙盒中选定的风格参数组合（详见 Style Engine 章节） |
| 参考文本 | 用户可选上传 | 否 | 已有章节、喜欢的作家作品，用于风格提炼和创意方向参考 |
| 项目初始偏好 | INIT 阶段 project.json | 是 | 目标总章数、最低单章字数、目标读者定位等基础约束 |

**交互流程：**
- Creative Director 驱动 CreativeOS 全部引擎，生成 3-5 个 Story DNA 变体并结构化展示
- 每个变体展示：核心矛盾设定、体裁融合方案、目标市场定位、四维新颖度评分明细
- 创意画布同步展示完整的 WhatIf 发散树，用户可在树中自由探索（点击展开、标记偏好、剪枝、以节点为新根重新发散、手动添加灵感节点、合并分支）
- 用户操作：选择变体 / 要求融合两个变体的优点 / 指定方向重新生成 / 深入某个创意分支
- 多轮讨论直到用户满意，所有中间创意和选择路径自动保存至 creative_path_history.json

### 5.3 STAGE 2: 世界观与角色工坊 【默认模式：discuss】

**输入：**

| 输入项 | 来源 | 必选 | 说明 |
|---|---|---|---|
| Story DNA | STAGE 1 story_dna.json | 是 | 核心矛盾、体裁融合方案、读者钩子、约束条件 |
| 创意路径历史 | STAGE 1 creative_path_history.json | 是 | 用户偏好标记、剪枝记录，帮助理解创作意图 |
| 概念设定 | STAGE 1 concept.json | 是 | 书名、基调、目标读者、主题列表 |

**2a. 世界观讨论：**
- Worldbuilder 基于 Story DNA 生成初始世界框架
- 用户与 Agent 讨论：势力格局是否合理、历史事件的逻辑自洽性、地理设定对剧情走向的影响
- 支持多轮调整直到满意

**2b. 角色成长工坊：**
- Character Designer 为每个核心角色生成完整档案和初始成长曲线方案
- 成长曲线包含：起点 → 关键转折点（此时仅绑定剧情事件类型，具体章节号在 STAGE 3 大纲确定后回填）→ 每次成长的代价（能力代价、心理代价、人际关系代价）→ 可能的低谷或倒退 → 终点
- 用户与 Agent 讨论成长节奏、代价充分性、与剧情高潮的同步程度
- 成长阶段与 StoryOS 的叙事资产（冲突、反转、揭秘）显式绑定

### 5.5 STAGE 4: 逐章写作循环 【默认模式：live】

**输入：**

| 输入项 | 来源 | 必选 | 说明 |
|---|---|---|---|
| 全部前阶段产出 | STAGE 1-3 | 是 | Story DNA、世界观、角色档案（含已回填章节的成长曲线）、大纲、伏笔映射、成长绑定 |
| 风格配置 | style_formula.yaml | 是 | 体裁模板 + 写作公式参数 + 禁忌约束列表 |
| 前置章节 | STAGE 4 已完成的各章 | 增量 | 第 N 章的写作依赖第 1 至 N-1 章的完整内容 |
| 叙事资产当前状态 | StoryOS 七项注册表 | 增量 | 每章更新后的当前状态 |
| 角色当前状态 | MemoryOS L0 | 增量 | 角色当前位置、身体状态、情绪状态、关系最新状态 |
| 读者指标当前值 | ReaderOS | 增量 | 上一章结束后的七项读者指标，用于本章节奏调控 |

**每章内部循环（六步）：**

**Step 1 — Scene 规划评审（用户可介入）：**
Writer 规划本章 3-6 个 Scene，每个 Scene 声明叙事目标和资产变化。用户可调整 Scene 顺序、增删 Scene、指定某 Scene 风格实验方向、为某 Scene 预设创新豁免。

**Step 2 — 逐幕写作 + 用户实时跟进：**
每幕写完后立即展示，不等整章完成。用户选择：接受 / 重写（附方向） / 自己修改（系统自动建议 SF_LOG 调整） / 标记偏好 / 申请创新豁免。

**Step 3 — 自动审查：**
Reviewer 执行三层审稿（详见一致性保障系统章节），不通过则打回重写（最多 3 次，第 3 次触发熔断降级通知用户）。

**Step 4 — 系统更新：**
StoryOS Agent 扫描 SF_LOG → 对比预声明 → 更新七类资产 → 级联传播（含冲突检测，见 StoryOS 章节）。MemoryOS L0/L1/L2/L4 增量更新。ReaderOS 重算七项指标。

**Step 5 — 章节评审会（用户可介入）：**
Reviewer 展示质量摘要，主动提问引导讨论。用户可快速通过或深入讨论。

**输出（每章累积产出）：**

`chapters/NNN_title.md` — 章节正文（含 SF_LOG 标记）
`review_report.json` — 每章审稿报告
`progress.json` — 进度追踪（持续更新）

StoryOS 七项注册表更新（每章写完后由 StoryOS Agent 自动更新）。

### 5.6 STAGE 5: 全书诊断与修订 【默认模式：review】

**输入：** 全部章节、全部审稿报告、进度追踪、大纲 + 伏笔映射、角色档案、叙事资产注册表当前状态。

**诊断维度与交互流程：**
系统自动扫描五大类问题，按 P0/P1/P2 分级展示。用户拥有最终决策权。

**重构区域专项诊断：**

> v1.4 新增。当项目存在 `restructuring_log.json`（即经历过叙事重构）时，STAGE 5 自动激活重构区域专项扫描。

专项扫描针对重构日志中标记的受影响章节，执行额外的一致性检查：

1. **新旧设定残留检测：** 扫描重构涉及章节中是否残留旧设定的痕迹（如旧版角色定位的对话、已废弃的能力规则描述）。对比重构前后的设定差异，在文本中搜索旧版关键词和表述模式
2. **重构边界连贯性：** 检查重构区域（P0/P1 章节）与非重构区域（P2 兼容章节）之间的叙事衔接是否平滑——角色状态、伏笔线索、冲突强度在边界处是否存在跳跃
3. **叙事资产对齐验证：** 验证重构日志中记录的叙事资产变更是否全部在 StoryOS 注册表中生效，没有遗留旧版资产状态
4. **重构完整性检查：** 逐一核验重构日志中 `completion_status` 为 `completed` 的修改项，确认对应的章节和资产确实已更新

## 六、协作机制详设

### 6.1 创意画布（Creative Canvas）

创意画布将 CreativeOS 的 WhatIf 发散树变成可交互的探索空间，替代传统"输入→输出"的黑盒模式。

用户在画布中的操作：点击展开、偏好标记（♡）、剪枝（✂）、以节点为新根发散（⟳）、手动添加节点（⊕）、合并分支（⊞）。每次操作后系统重新计算各节点的新颖度评分。用户满意后选定一条路径，系统据此生成最终 Story DNA。产出还包括创意路径选择历史。

### 6.2 分支模拟引擎（Branch Simulation）

> v1.3 增强：标注 LLM 推理与确定性计算的边界，增加置信度提示。

分支模拟引擎让用户在情节设计阶段可以安全地探索"如果我改了这个会怎样"，而无需真的修改大纲。

**确定性计算部分（无需 LLM，结果可复现）：**

以下分析由 Conductor 基于结构化数据直接计算：

| 分析项 | 计算方式 |
|---|---|
| 受影响的章号范围 | 遍历 outline.json 的章节序列和伏笔映射表，计算依赖闭包 |
| 受影响的角色列表 | 扫描受影响章节的 Scene 规划中的角色引用和 SF_LOG 关联 |
| 受影响的伏笔列表（结构层面） | 对比新旧大纲中伏笔映射表的 planted_chapter、clue_chapters、planned_reveal_chapter 字段变化 |
| 成长曲线章节偏移 | 对比修改前后 growth_bindings 中 turning_point 的 chapter 字段变化 |

**LLM 推理部分（需置信度标注，结果供参考）：**

| 分析项 | 推理方式 | 置信度标记 |
|---|---|---|
| 张力曲线的预期变化趋势 | LLM 基于修改前后的叙事资产分布预测 | 🟡 中置信度 — 基于叙事资产数量变化的方向性预测，非精确值 |
| 哪些伏笔"可能断裂" | LLM 评估伏笔与修改后上下文的语义兼容性 | 🟡 中置信度 — 语义评估，非确定性 |
| 替代方案建议 | LLM 生成修改方案的变体 | 🟠 低置信度 — 创意性建议，仅供用户参考 |
| 读者体验指标的预估变化 | LLM 综合评估 | 🟡 中置信度 — 基于模式类比，非精确计算 |

```
│  [🟡 LLM推算] 张力曲线变化：               │
│    修改前：___/‾‾‾\___                    │
│    修改后：__/‾‾‾‾‾‾\__ （更早进入高潮）   │
│    置信度：中（方向性预测，非精确值）        │
│                                          │
│  [🟡 LLM推算] 可能断裂的伏笔：             │
│    fs_005（师父身份线索密度不足，建议       │
│    在第15-19章补充至少1条线索）             │
│                                          │
│  [🟠 LLM建议] 替代方案：                   │
│    · 仅提前3章（到第27章），影响范围更小     │
│    · 拆分为两次揭示：第20章部分揭示 +        │
│      第30章完整揭示                        │
│                                          │
│  [应用此修改] [对比替代方案] [取消]         │
└─────────────────────────────────────────┘
```

用户在依据 LLM 推理结果做决策时，系统会提醒："此结论基于 AI 推理，置信度为中等。建议结合你的创作直觉做最终决策。"

### 6.3 成长工坊（Growth Workshop）

成长工坊将角色设计从"填写档案"升级为"协同设计成长曲线"。每个核心角色的成长曲线包含起点、关键转折点（绑定剧情事件）、低谷/倒退、终点。Character Designer 生成初始方案后，用户与其深入讨论节奏、代价和同步性。系统以可视化方式展示成长曲线图，修改后自动检查与剧情里程碑的一致性。

### 6.4 风格沙盒与创新豁免

**风格沙盒：** 写作前用户用约 500 字测试文本预览不同风格参数组合的效果，可混合多个版本的参数形成自定义风格配置。

**创新豁免：** Writer 可主动提交豁免申请（声明突破规则、创作意图、预期效果），用户审批通过后该 Scene 不受对应规则约束。效果优秀可提炼为新规则，效果不佳标记为反例。

### 6.5 章节评审会（Chapter Review）

每章完成后，Reviewer 展示质量摘要和开放问题引导讨论。用户可标记偏好、提出修改意见、记录灵感，或选择"快速通过"（< 30 秒）。

### 6.6 灵感捕捉（Inspiration Capture）

灵感路由器在所有阶段的讨论中持续运行，自动识别、分类、存储灵感。阶段切换时主动提醒用户回顾。用户可在灵感面板中拖拽灵感应用到当前阶段。

### 6.7 用户编辑辅助

用户在 `live` 模式下手动修改 Scene 文本后，系统自动分析编辑内容中隐含的叙事变化，建议需要添加/调整的 SF_LOG 标记，确保手动改写也能被 StoryOS 正确追踪。

### 6.8 叙事重构模式

> v1.3 新增。针对审查指出的"系统越精确，偏离成本越高"的结构性张力，提供批量处理大胆偏离的机制。

当用户在第 N 章（通常在写作中后期）提出一个**全局性的创意偏离**——即改变一个在早期阶段已确定的核心设定——逐条确认影响的成本过高。叙事重构模式提供批量处理能力。

**触发条件（满足任一）：**

- 用户明确声明"我想要一个大的情节转向"
- 用户修改了 STAGE 1-3 的产出物且影响范围 ≥ 10 章
- 用户在章节评审会中连续 3 章标记"此方向需要大幅调整前文"

**模式流程：**

**Step 1 — 意图声明：**

用户用自然语言描述偏离意图：
```
"我想让苏晓晓从一开始就是代码观察者的核心成员，
而不是第58章才揭示。她的背叛应该是假的——
她一直在暗中保护林峰。"
```

**Step 2 — 全量影响推算：**

Conductor 联合 StoryOS 和 Outliner，推算此偏离对全部已有产出的影响：

```
┌─────────────────────────────────────────────┐
│          叙事重构影响分析                     │
│                                              │
│  重构声明：苏晓晓角色定位从"监视者→背叛者"     │
│           改为"核心守护者（伪装监视者）"       │
│                                              │
│  直接影响：                                   │
│  ├─ characters.json：苏晓晓角色档案需重写      │
│  │   - 核心人格层：desires 从"完成观察任务"     │
│  │     改为"保护林峰不被组织发现"              │
│  │   - 成长曲线：tp_002 从"背叛"改为"身份暴露"  │
│  ├─ outline.json：第58章高潮事件需重设计       │
│  └─ world.json：代码观察者 faction 需调整      │
│                                              │
│  章节影响：                                   │
│  ├─ P0 必须重写（3章）：第48/58/82章           │
│  │   苏晓晓的关键转折场景                       │
│  ├─ P1 建议调整（12章）：第7-47章中            │
│  │   苏晓晓与林峰的互动需要调整语气和潜台词      │
│  └─ P2 兼容（85章）：无苏晓晓出场或出场         │
│      方式与新设定不冲突                        │
│                                              │
│  叙事资产影响：                                │
│  ├─ Twist tw_005 从"背叛揭示"改为"身份揭示"    │
│  ├─ Conflict cf_007 需重新定义                 │
│  ├─ 新增 Mystery：苏晓晓的真实目的何时暴露？    │
│  └─ 级联影响：关联的 Reveal/Expectation        │
│      共 7 项需更新                             │
│                                              │
│  预估工作量：P0 3章重写 + P1 12章修订           │
│              ≈ 额外 15-20 章写作时间            │
│                                              │
│  [执行重构]  [缩小范围（仅调整部分）]  [取消]    │
└─────────────────────────────────────────────┘
```

**Step 3 — 批量执行：**

用户确认后，系统按优先级批量处理：

1. 更新所有受影响的产出物（world.json、characters.json、outline.json）
2. 标记受影响章节的状态（`pending_rewrite` / `pending_revision` / `compatible`）
3. P0 章节进入 STAGE 4 的重写队列，在每个 Scene 开始前注入"重构上下文"——新旧设定的差异摘要
4. P1 章节在章节评审会中附带"重构对齐检查"——Reviewer 额外检查这些章节与新设定的兼容性
5. StoryOS 批量更新叙事资产（含级联冲突检测，见 StoryOS 章节）

**Step 4 — 重构追踪：**

系统生成一份 `restructuring_log.json`，记录本次重构的完整信息：
- 原始设定 vs 新设定的差异
- 受影响的文件和章节清单
- 每项修改的完成状态
- 重构前后连贯性评分的对比

这份日志在 STAGE 5 全书诊断时被特别关注——重构区域是诊断的重点扫描区。

**与传统回退的区别：**

| | 传统回退（2.2 节） | 叙事重构模式 |
|---|---|---|
| 触发方式 | 用户主动回退到之前阶段 | 用户在写作中声明全局偏离意图 |
| 影响粒度 | 逐项确认 | 批量推算 + 用户审阅 |
| 适用场景 | 局部修改（影响 < 10 章） | 全局偏离（影响 ≥ 10 章） |
| 决策方式 | 逐条决策 | 整体决策 + 分级执行 |

## 七、跨作品资产复用

> v1.3 新增。v1.2 中只有 Style Engine 的风格模板支持跨项目复用，Trope Pool、Idea Pool、Contradiction 模板等都是项目锁定的。本章建立统一的跨作品资产复用机制。

### 7.1 设计目标

高频写作的作者在写完第一部作品后，不应从零开始第二部。系统在项目间共享三类资产：市场知识（Trope Pool）、创意积累（Idea Pool、Contradiction 模板）、风格资产（Style 模板）。共享范围分为三级：用户私有（默认）、团队共享（协作场景）、公共匿名（社区贡献）。

### 7.2 资产共享范围

| 资产类型 | 存储层级 | 共享范围 | 更新机制 |
|---|---|---|---|
| **Global Trope Pool** | 用户级 | 用户所有项目自动共享 | 人工标注 + 季度市场数据爬取 + 用户标记（"这个套路最近很火/过气了"） |
| **Global Idea Pool** | 用户级 | 用户所有项目自动共享 | 灵感路由器跨项目捕捉 + 用户手动添加 + 项目灵感可"提升"为全局灵感 |
| **Contradiction Templates** | 系统级（内置 5 个）+ 用户级（自定义） | 内置模板随版本更新；用户自定义模板存储在 `~/storyforge/global/contradiction_templates.json`，跨项目共享 |
| **Style Formula Templates** | 用户级 | 可导出为 `.style.yaml` 文件 | 用户保存自定义体裁模板 → 自动出现在其他项目的体裁选择列表中 |
| **Genre Fusion Rules** | 系统级 | 内置兼容/冲突矩阵 + 用户可扩展 | 系统版本更新内置矩阵；用户可添加"自定义体裁"及其兼容规则 |
| **SF_LOG 模式库** | 用户级 | 跨项目 | 用户标记为"优秀"的 Scene → 其 SF_LOG 模式被提取为参考模板，供新项目的 Writer 参考 |

### 7.3 Global Trope Pool

用户级的套路模式知识库，跨项目共享：

```json
{
  "trope_id": "trp_001",
  "name": "重生",
  "core_mechanism": "主角死亡后回到过去某一时间点，利用前世记忆/知识改变命运",
  "market_saturation": {
    "current": 0.72,
    "updated_at": "2026-Q2",
    "trend": "declining",
    "source": "manual_review"
  },
  "sub_variants": [
    {"name": "签到重生", "saturation": 0.85},
    {"name": "反派重生", "saturation": 0.45},
    {"name": "群穿重生", "saturation": 0.30}
  ],
  "fusion_potential": ["mystery", "political", "horror"],
  "anti_patterns": [
    "重生后立刻无敌——缺乏成长弧线",
    "前世记忆仅用于打脸——浪费重生设定的叙事深度"
  ],
  "projects_used_in": ["proj_001", "proj_003"],
  "user_notes": "2024-2025大热，2026年有所降温但反派重生和群穿变体仍是蓝海"
}
```

用户在 INIT 阶段和 STAGE 1 可以浏览 Global Trope Pool，了解市场趋势。在 STAGE 1 的创意画布中，WhatIf 发散会参考 Trope Pool 的 market_saturation 数据——高饱和度方向的节点标注"[红海]"警告，低饱和度方向标注"[蓝海]"推荐。

市场饱和度数据的更新来源：
- **系统内置快照：** StoryForge 每个大版本附带一份市场饱和度快照（由开发团队基于公开网文数据维护），作为 Global Trope Pool 的初始值和更新来源。快照随版本升级自动合并到用户本地数据，不覆盖用户自行标注的调整
- **用户反馈：** 用户在灵感面板中标记"最近这个方向很火/降温了"，系统记录后微调权重。用户标注的优先级高于系统快照——如果用户标记某套路为"已过气"而快照中仍为低饱和度，以用户标注为准
- **项目内统计：** 如果用户某类套路连续使用多次，系统提示"此套路在你的作品中已出现 X 次"

### 7.4 Global Idea Pool

用户级的灵感库，跨项目收集和复用：

```
灵感来源：
├── 项目灵感路由器捕捉（自动同步到 Global Pool）
├── 用户手动添加（在任何时候记录的"想写但还没写的点子"）
├── 项目完成后"提升"——用户将项目中未使用但有价值的灵感提升为全局灵感
└── 阅读/观影捕捉——用户手动记录的外部灵感
```

Global Idea Pool 与项目级 Idea Pool 的关系：
- 项目创建时，可以从 Global Idea Pool 中选择相关灵感"导入"项目
- 项目进行中产生的灵感默认存入项目级 Idea Pool；用户可手动"提升"到 Global Pool
- 项目完成后，系统提示："本作品产生了 X 条灵感，其中 Y 条未使用。是否将其中有价值的灵感保存到全局灵感库？"

### 7.5 跨项目风格模板

用户在风格沙盒中微调出的自定义风格配置，可保存为命名模板：

```yaml
# ~/storyforge/templates/styles/dark_sci_fi_mystery.style.yaml
name: "暗黑科幻悬疑风"
based_on: cool_novel
overrides:
  avg_sentence_length: {min: 18, max: 30}  # 比标准爽文略长
  dialogue_ratio: 0.35
  description_density: 0.25  # 环境描写占比更高
  hook_style: "cliffhanger_preferred"
  protagonist_taboos:
    - "禁止主角在非关键场景使用能力"
    - "记忆侵蚀的描写必须具体（哪段记忆、什么情感）"
```

该模板保存后，在新项目 INIT 阶段的 Step 3（风格方向初选）中出现在"我的自定义模板"列表中。

## 八、Style Engine — 风格引擎

### 8.1 设计目标

AI 写作容易"千篇一律"。Style Engine 让作者能够定义和控制作品的风格，从宏观流派模板到微观句式规则。通过风格沙盒和创新豁免为创新写作留出用户可控的空间。

### 8.2 三层风格控制

**L1 体裁模板（Genre Templates）：**

预定义的流派写作规则，以 YAML 外置，体裁可插拔。系统内置五种体裁模板（爽文、严肃文学、悬疑推理、科幻、奇幻），用户可通过跨项目风格模板（第七节）创建和复用自定义体裁。

**L2 写作公式（Writing Formulas）：**

句子级别的定量约束：句式模板、情绪节奏模板、对白节奏公式、爽点密度公式、环境描写公式。

**L3 禁忌约束（Constraint Layer）：**

分级别的"不能做"规则：全局禁忌、角色禁忌、体裁禁忌。

### 8.3 风格提炼（Style Extractor）

作者上传参考文本，系统自动分析句式、节奏、词汇、结构特征，生成可复用的 YAML 规则文件。提取结果可在风格沙盒中预览，也可保存为跨项目模板。

### 8.4 风格沙盒

写作前用约 500 字测试文本预览不同参数组合的效果，可混合多版本参数。写作中也可随时回到沙盒测试调整后的参数效果。

### 8.5 创新豁免

Writer 提交豁免申请（声明突破规则、创作意图、预期效果）→ 用户审批 → 效果评估 → 优秀可提炼为新规则，不佳标记为反例。

## 九、CreativeOS — 创意引擎

### 9.1 设计目标

帮助作者从一个模糊的想法出发，系统化地发散出高新颖度、有市场潜力的故事设定。CreativeOS 与创意画布深度集成，用户是创意探索过程的主动参与者。

### 9.2 七大创意模块

**灵感种子库（Idea Pool）：**

存储和管理所有创意灵感。项目级 Idea Pool 在项目内运作，灵感可提升到 Global Idea Pool（第七节）供跨项目复用。

**套路模式库（Trope Pool）：**

系统化的网文套路知识库。项目创建时从 Global Trope Pool（第七节）继承当前市场数据。项目写作期间，Trope Pool 在 STAGE 1 被 CreativeOS 引用，在创意画布中影响 WhatIf 发散方向和新颖度评分。

**套路变异器（Mutation Engine）：**

四种变异操作：逆转（Inversion）、融合（Fusion）、升维（Escalation）、打破（Subversion）。每次变异输出核心设定、核心矛盾、新奇钩子、自洽性检查。

**矛盾设定生成器（Contradiction Engine）：**

五个内置矛盾模板（能力×限制、永恒×消逝、身份×秘密、目标×代价、力量即弱点），随系统版本更新。用户可基于内置模板创建自定义矛盾模板，自定义模板自动保存到用户级模板库（`~/storyforge/global/contradiction_templates.json`，第七节），跨项目可用。

**连续发散器（WhatIf Engine）：**

从核心前提出发递归发散，默认深度 3 层、广度 4 个分支，最多 84 个衍生节点。约束条件：每个子节点必须是父节点的直接逻辑推论。

**体裁融合器（Genre Fusion Engine）：**

在结构特征层面融合两个体裁。体裁兼容性矩阵内置，用户可扩展自定义体裁。体裁距离通过 BFS 计算，距离越远新颖度加成越大。

**新颖度评估器（Novelty Evaluator）：**

四维评估（市场饱和度 30%、套路相似度 25%、矛盾深度 25%、讨论潜力 20%）。综合判定：≥80 高新颖度可推进 | ≥60 中等建议强化矛盾 | ≥40 偏低需显著变异 | <40 建议更换核心创意。

爆款潜力 = 新颖度(40%) + 反转密度(30%) + 讨论潜力(30%)。

### 9.3 创意画布交互层

创意画布将 WhatIf 树可视化为可交互探索空间（详见 6.1 节）。

### 9.4 产出物

Story DNA（故事基因）+ 创意路径选择历史。

## 十、StoryOS — 叙事资产管理

### 10.1 设计目标

将叙事中的"无形资产"——冲突、承诺、谜团、反转、揭晓、目标、读者期待——显式登记、追踪、管理。每一个伏笔都有生命周期，每一条剧情线都有始有终。与分支模拟引擎集成，支持在大纲阶段推演连锁影响。

### 10.2 七类叙事资产

| 资产类型 | 追踪的核心问题 | 关键属性 |
|---|---|---|
| **冲突（Conflict）** | 谁和谁在斗？斗到什么程度了？ | 类型、所有者、目标、当前强度、创建章节、预期解决章节、升级历史、解决说明 |
| **承诺（Promise）** | 角色对谁许下了什么？什么时候兑现？ | 内容、说话人、听众、创建章节、截止章节、重要性、状态、兑现说明 |
| **谜团（Mystery）** | 什么悬念吊着读者？线索够了吗？ | 问题、埋设章节、已给出线索列表、计划揭示章节、状态、揭示后影响级别 |
| **反转（Twist）** | 什么颠覆在铺垫？读者现在预期什么？ | 类型、描述、铺垫起始章节、铺垫方式、计划揭示章节、揭示触发条件、影响级别、状态 |
| **揭晓（Reveal）** | 什么秘密要揭示？谁知道/不知道？ | 秘密内容、已知角色列表、读者是否已知、计划揭示章节、揭示方式、状态 |
| **目标（Goal）** | 角色要达成什么？进展如何？ | 所有者、类型、内容、创建章节、目标章节、状态、子目标列表 |
| **读者期待（Expectation）** | 读者现在在期待什么发生？ | 类型、内容、植入章节、兑现章节、兑现方式、状态 |

### 10.3 叙事资产关联网络

这些资产不是孤立的，它们之间存在天然的因果关系。系统维护跨资产的外键引用（详细关联图谱同 v1.2）。

### 10.4 级联传播与冲突检测

> v1.3 增强：新增级联冲突检测与隔离机制。

**正常级联规则：**

当一处资产状态变化时，系统自动级联传播更新关联资产：

- Mystery 状态变为 `revealed` → 关联的 Reveal 自动变为 `revealed` → 关联的 Expectation 自动变为 `fulfilled`
- Twist 状态变为 `revealed` → 关联的 Expectation 自动变为 `ready_to_fulfill`
- Reveal 状态变为 `revealed` → 关联的 Conflict 自动变为 `escalated`

**级联前校验（每次级联传播前自动执行）：**

1. **循环依赖检测：** 遍历级联传播路径，检测是否存在 A → B → ... → A 的循环。发现循环时，中断级联，标记涉及的资产为"循环依赖——需人工解耦"。

2. **状态冲突检测：** 在更新每个关联资产前，检查新状态是否与当前状态逻辑兼容：

| 当前状态 | 拟更新状态 | 兼容性 | 处理 |
|---|---|---|---|
| `resolved` | `active` | ❌ 冲突 | 已解决的资产不应被重新激活（除非通过叙事重构模式） |
| `revealed` | `foreshadowing` | ❌ 冲突 | 已揭示的 Twist 不应退回铺垫状态 |
| `fulfilled` | `accumulating` | ❌ 冲突 | 已兑现的 Expectation 不应被重新积累 |
| `abandoned` | 任何状态 | ⚠️ 警告 | 用户主动放弃的资产被级联触发——可能是无意的，需确认 |

3. **互斥资产冲突检测：** 检查同一个 Mystery 是否关联了两个状态互斥的 Twist（如一个已 `revealed` 而另一个才 `foreshadowing`，且两者的揭示条件互相依赖）。检测到时，标记为"潜在逻辑冲突"，不阻断级联但生成警告，在下一章 Scene 规划时提醒 Writer。

**冲突处理流程：**

```
级联传播触发
    │
    ├── 循环依赖？ → 是 → 中断级联，标记资产，通知用户
    │
    ├── 状态冲突？ → 是 → 中断该条级联链，保留其他链继续传播
    │                   标记冲突资产 → 章节评审会中展示 → 用户决策
    │
    └── 互斥冲突？ → 是 → 不阻断级联，生成警告标记
                         章节评审会中展示 → Writer 注意避开矛盾
```

**手动干预入口：**

当级联因冲突而中断时，用户有三种处理选项：
- **解耦资产关联：** 移除导致冲突的跨资产引用关系
- **强制覆盖：** 用户确认新的资产状态是正确的，忽略冲突（标记为 `force_updated`）
- **叙事重构：** 如果冲突的根源是更大的设定问题，进入叙事重构模式（6.8 节）

### 10.5 全局伏笔映射表

从大纲规划阶段就建立伏笔的完整生命周期追踪。每个伏笔追踪其状态、关联角色、受影响的故事资产。在分支模拟中，系统自动分析修改对伏笔映射表的影响。

### 10.6 全书张力曲线

系统实时计算整本书的张力值（0-100），综合当前活跃叙事资产状态。包含智能预警（张力过低、承诺逾期、期待超期、冲突不足、谜团无进展）。

## 十一、SF_LOG 变化标记机制

### 11.1 设计目标

传统 AI 写作中叙事变化散落在自然语言中，难以精确识别。StoryForge 的做法是：写作时 Writer 在叙事文本中嵌入结构化标记（Markdown 注释语法，对读者不可见），系统用正则确定性解析，零 LLM 调用，精确、确定、零遗漏。

### 11.2 变化标记类型

**角色状态变化：** `character_relation_change`、`character_emotion`、`character_location_change`、`character_physical_change`

**叙事资产变化：** `knowledge_gain`、`conflict_escalate`、`mystery_clue`、`twist_reveal`、`expectation_fulfill`、`goal_milestone`、`registry_create`

### 11.3 工作流程

**写作前：** Writer 在 Scene 规划时填写 `registry_changes`（预声明）和 `required_logs`（必须嵌入的标记类型）。

**写作中：** Writer 在叙事文本的合适位置嵌入变化标记。

**写作后：** StoryOS Agent 用正则解析所有标记（确定性）。将解析结果与 Writer 的预声明对比——匹配成功的变化被验证通过并应用；预声明了但未找到对应标记的变化被记录为"遗漏"并**阻断**（Fact Guard 第 5 项硬检）；出现了但未预声明的变化被接受（Writer 在写作中主动新增的变化）。

**用户编辑后：** 当用户在 `live` 模式下手动修改文本后，系统自动分析修改内容中隐含的叙事变化，向用户建议需要添加或调整的 SF_LOG 标记。

### 11.4 语义完整性预检

> v1.3 新增。针对审查指出的 SF_LOG 可靠性风险——Fact Guard 第 5 项硬检覆盖了"预声明标记是否缺失"，但 Writer 未预声明也未标记的新增叙事变化仍是盲区。语义完整性预检在 Fact Guard 之前运行，弥补这个盲区。

**预检范围：**

仅对高重要性叙事事件类型执行语义扫描（Tier 3 模型，低开销）：

| 事件类型 | 检测方式 | 触发条件 |
|---|---|---|
| `twist_reveal` | LLM 语义扫描 | 检测 Scene 文本中是否出现了重大信息颠覆（角色身份、核心设定、关系性质的揭示）但未嵌入对应标记 |
| `registry_create` | LLM 语义扫描 | 检测 Scene 文本中是否创建了新的冲突线/承诺/谜团但未嵌入对应标记 |
| `character_relation_change` | 关键词 + 模式匹配 | 检测 Scene 文本中是否出现了关系状态词（"反目""决裂""和解""结盟"）但未嵌入对应标记 |

**预检输出（不阻断，仅建议）：**

预检发现的潜在遗漏以"建议补充标记"的形式展示，**不阻断** Scene 的通过——因为它基于 LLM 语义判断，存在误报可能：

```
┌─────────────────────────────────────────┐
│  语义完整性预检                           │
│                                          │
│  ⚠️ 检测到 2 处可能需要 SF_LOG 标记：      │
│                                          │
│  1. 第 3 幕第 45 行附近：                 │
│     "林峰突然意识到眼前这个人不是他认识     │
│      的苏晓晓"                             │
│     可能涉及：character_relation_change    │
│     建议标记：[添加标记] [忽略]             │
│                                          │
│  2. 第 3 幕第 78 行附近：                 │
│     "这是林峰第一次发现能力可以被反向追踪"   │
│     可能涉及：knowledge_gain               │
│     建议标记：[添加标记] [忽略]             │
│                                          │
│  [全部忽略] [逐条处理]                     │
└─────────────────────────────────────────┘
```

**与 Fact Guard 的关系：**

| | Fact Guard 第 5 项 | 语义完整性预检 |
|---|---|---|
| 检查对象 | `required_logs` 中**预声明**的标记是否在文本中出现 | **未预声明**的叙事变化是否遗漏了标记 |
| 检查方式 | 确定性（正则匹配） | LLM 语义扫描（Tier 3 模型） |
| 阻断性 | **阻断** | **不阻断**（建议性质） |
| 误报风险 | 零（确定性） | 存在（LLM 语义判断） |

语义完整性预检的设计原则：**确定性检查做阻断；概率性检查做建议。** 不对 LLM 的判断赋予阻断权力，避免误报伤害创作流。

**开销估算：**

单 Scene 预检扫描约 500 tokens（Tier 3 模型，约 $0.00005/scene）。以 100 章 × 平均 4 Scene = 400 Scene 计，全书累计约 200K tokens，总成本 < $0.02，占 STAGE 4 总 token 预算的 < 0.2%。用户可在配置中关闭预检以消除此开销。

## 十二、Scene Engine — 写作执行系统

### 12.1 设计目标

将叙事资产的规划落实到每一幕的写作。每个 Scene 带有明确的叙事任务和读者体验目标。支持用户实时介入、创新豁免申请和用户编辑辅助。

### 12.2 Scene 的叙事规格

每个 Scene 在写作前需要明确三个层面的规划：

**叙事层：** 本幕目标、本幕冲突、情感弧线、叙事角色（小爽点/大爽点/铺垫/过渡/悬念结尾）。

**读者体验层：** 是否有反转、兑现了什么、结尾钩子、预期读者反应。

**叙事资产管理：** `registry_changes`（本幕涉及哪些叙事资产的状态变化）、`required_logs`（本幕必须嵌入哪些类型的 SF_LOG 标记）。

### 12.3 Scene 规划评审（用户介入点）

每章开始前 Writer 展示 Scene 规划供用户审阅。用户可调整顺序、增删 Scene、指定风格实验方向、预设创新豁免。

### 12.4 逐幕写作与用户实时跟进

默认 `live` 模式下每幕写完后展示。用户可接受、要求重写（附方向）、自己修改（系统自动建议 SF_LOG 标记）、标记偏好。

### 12.5 写作规范约束

通用写作规范（可按体裁覆盖）：每章至少 4000 字、每章结尾必须有钩子、禁止"吃书"、禁止元引用、爽文体裁特化规则。

## 十三、MemoryOS — 五层记忆系统

### 13.1 设计目标

长篇写作中，不同时效的信息需要用不同方式管理，精准召回而不污染 AI 的上下文窗口。五层记忆按检索优先级从近到远排列：L0 → L1 → L4 → L2 → L3。

### 13.2 五层结构

**L0 运行时记忆（约 500 tokens，始终在上下文顶部）：** 当前章的即时状态快照。

**L1 热记忆（约 15K tokens）：** 最近 5 章的完整内容，每 5 章触发一次关键细节重提取。

**L2 温记忆（约 8K tokens）：** 全书的结构化摘要——按卷/章组织的摘要树、角色关系图、全书时间轴、伏笔列表。

**L3 冷记忆（按需召回）：** 全书内容的向量化语义检索库。混合检索策略：向量语义相似度 + BM25 关键词精确匹配，RRF 融合排序。

**L4 叙事记忆（约 3K tokens）：** 与 StoryOS 强同步的叙事资产摘要。

### 13.3 细节防遗忘机制

除 L1 的每 5 章关键细节重提取外，系统维护"活跃物品与线索"追踪表——记录了角色持有但尚未使用的物品、角色受过的伤、未完成的对话约定。这些信息在写作时被注入 Writer 的上下文。

## 十四、ReaderOS — 读者体验模型

### 14.1 设计目标

作者需要知道读者的阅读体验会是什么样。ReaderOS 从七个维度量化每章的读者体验，全部基于可计算的指标，不依赖 AI 的"感受判断"。

### 14.2 七个读者状态指标

| 指标 | 衡量什么 | 计算逻辑 |
|---|---|---|
| **好奇心（Curiosity）** | 读者有多想知道接下来发生什么？ | 当前所有开放谜团 × 各自影响权重求和（世界观级=30、剧情级=20、角色级=10），归一化到 0-100 |
| **张力（Tension）** | 读者有多紧张？ | 直接引用 TensionCurve 的综合计算值 |
| **满足感（Satisfaction）** | 读者最近爽了几次？ | 近 3 章内 fulfilled 状态的 Expectation 和 Promise 数量 × 20，上限 100 |
| **挫败感（Frustration）** | 主角是不是太惨了？ | 近 5 章内主角 Goal 被阻断或失败的次数 × 15，上限 100 |
| **疲劳度（Fatigue）** | 读者是不是太累了？ | 近 3 章 Tension 平均值减去 50（超过 50 视为高强度），超额部分 × 1.5 |
| **追更欲（Addiction）** | 读者会不会点下一章？ | 好奇心×0.30 + 张力×0.25 + 满足感×0.20 + 结尾钩子质量×0.25 |
| **讨论潜力（Discussion Potential）** | 读者会不会在评论区吵架？ | 道德模糊度 + 预期打破 + 结尾钩子强度×0.25 + 文本争议特征 |

### 14.3 阈值基准说明

> v1.3 新增。v1.2 中的预警阈值没有说明来源。本节交代基准依据和体裁差异化。

**基准来源：**

ReaderOS 的预警阈值来自三个来源的综合校准：

| 来源 | 说明 | 贡献 |
|---|---|---|
| **行业经验值** | 基于网文行业（起点、番茄、晋江等平台）的读者行为数据——追读率、弃书率、章节评论量——映射到叙事特征后反推 | 追更欲、疲劳度、好奇心的核心阈值范围 |
| **内部测试** | 在 StoryForge 开发过程中，使用 5 部不同体裁的完本网文回测 ReaderOS 指标，校准阈值 | 各项阈值的精确数值和梯度 |
| **用户自定义** | 作者在配置中覆盖默认阈值（见 18.4 节） | 不同作者的创作风格偏好 |

阈值的设计原则是**偏保守**——宁可多提醒一次，也不要漏过一个可能导致读者流失的信号。所有阈值均为默认值，用户可在配置中调整。

### 14.4 体裁差异化阈值

> v1.3 新增。不同体裁的"正常"读者体验差异显著，统一阈值会造成无效告警。

| 指标 | 爽文（Cool Novel） | 严肃文学（Literary） | 悬疑推理（Mystery） | 科幻（Sci-Fi） | 奇幻（Fantasy） |
|---|---|---|---|---|---|
| **追更欲 < 阈值（严重）** | 50 | 30 | 45 | 35 | 40 |
| **挫败感 > 阈值（高度）** | 60 | 80 | 65 | 70 | 70 |
| **疲劳度 > 阈值（中度）** | 55 | 40 | 60 | 55 | 50 |
| **好奇心 < 阈值且追更欲 > 50（中度）** | 35 | 20 | 30 | 25 | 25 |
| **张力 > 阈值但讨论潜力 < 30（轻度）** | 55 | 70 | 60 | 65 | 60 |

**差异化逻辑：**
- 爽文读者的追更欲阈值更高（50），因为爽文的核心竞争力就是"让人想点下一章"。追更欲降到 50 以下对爽文是危险信号，但对严肃文学可能只是正常波动
- 严肃文学的挫败感容忍度更高（80），因为严肃文学中主角受挫是叙事常态
- 悬疑推理的疲劳度阈值更高（60），因为悬疑本身需要持续的认知投入，读者预期就是"累但欲罢不能"

体裁差异化阈值在创建项目时根据选定的体裁自动配置，用户可在配置文件中针对具体阈值进行覆盖。

### 14.5 自动预警

系统在指标触发阈值时自动生成分级预警，预警同时注入下一章的写作提示。各项预警的阈值说明和体裁差异见 14.3-14.4 节。

## 十五、一致性保障系统

### 15.1 角色一致性

每个重要角色维护四层档案：核心人格层、当前状态层、能力进化追踪、角色声音签名。信念变化有严格门槛：必须来自 SF_LOG 标记、需要 ≥ 2 个独立触发事件（近 3 章内）、≥ 1 个在本章、触发类型在白名单内。

### 15.2 剧情线一致性

每条剧情线有完整生命周期：创建 → 激活 → 升级 → 解决（或放弃）。系统自动检测长期未推进的剧情线。

### 15.3 三层审稿机制

每幕写完后，Reviewer 自动执行三层审查：

**第一层·事实核查（Fact Guard，硬性阻断）：**

六项检查（前五项为纯规则匹配，零 LLM 调用；第 6 项为预检结果的确定性复核，预检本身使用 Tier 3 LLM 但复核逻辑不调用 LLM）：

1. **时间线连续性：** 角色不能同时出现在两个位置。同章内位置变化需要对应的 SF_LOG 标记说明位移
2. **角色状态一致性：** 角色不能展示不该知道的秘密。角色不能做出 voice_signature 中 `forbidden` 列表禁止的行为
3. **世界规则一致性：** 能力使用不超过当前阶段上限。突破需有对应的代价 SF_LOG 标记
4. **叙事资产合规：** 不能将已解决的条目直接改回 active。引用的 Registry 条目必须存在且状态兼容
5. **变化标记完整性：** `required_logs` 中声明的标记是否全部在文本中出现。SF_LOG 标签的格式是否正确（严格正则校验）
6. **语义完整性预检结果复核：** > v1.3 新增。检查 Scene 文本中是否存在高重要性的未标记叙事变化（由 11.4 节语义完整性预检识别）。预检结果作为第 6 项检查的输入——如果预检发现了高置信度的遗漏，Fact Guard 列出但**不阻断**（因为预检基于 LLM 语义判断，存在误报可能）。用户在 Scene 通过前需确认"已审阅预检建议"——可以全部忽略，但必须确认

不通过时，自动打回重写（最多 3 次），每次附带具体修改建议。第 3 次仍不通过时触发熔断降级。

**第二层·叙事建议（Narrative Guard，软性建议，不阻断）：**

检测节奏、情绪连贯性、角色行为动机。包含状态漂移检测——检查 Scene 文本中是否有明显的行为变化但没有对应的 SF_LOG 标签。如果发现漂移，生成 warning 建议（不是阻断），让用户在章节评审会中注意。

**第三层·风格标签（Style Guard，仅记录）：**

检测是否触犯角色或流派的禁忌规则。触碰记录被标记但不阻断。创新豁免范围内的规则不再检测。

### 15.4 连贯性评分

每章完成后综合评估质量（0-100 分）。低于用户设定的阈值（默认 80）自动触发重写：

| 维度 | 权重 | 扣分规则 |
|---|---|---|
| 时间线连续性 | 25% | 每处断层 -5 分 |
| 人物状态一致性 | 25% | 每处矛盾 -10 分 |
| 伏笔完整性 | 20% | 每个遗漏/意外 -5 分 |
| 世界规则一致性 | 20% | 每处违规 -8 分 |
| 禁忌约束 | 10% | 每处违禁 -15 分（创新豁免除外） |

## 十六、多轮修订流水线

### 16.1 全书自动诊断

系统对已完成的所有章节进行一键质量诊断，分五大类问题：时间线断裂、角色不一致、节奏问题、重复冗余、伏笔追踪。

### 16.2 分级修复 + 用户决策

采用"系统诊断 → 分级展示 → 用户决策"：

- **P0 阻断级：** 系统强烈建议修复，但由用户确认或主动跳过
- **P1 质量级：** 用户逐项决定是否修复
- **P2 优化级：** 用户选择性修复

每级问题附带修复策略建议，最终决策权在用户手中。

## 十七、Agent 协作体系

StoryForge 有七个各司其职的 Agent，每个 Agent 在对应的创作阶段与用户协作：

| Agent | 职责 | 工作阶段 | 协作模式 | 产出物 |
|---|---|---|---|---|
| **Creative Director** 创意总监 | 驱动 CreativeOS 全部引擎，在创意画布中与用户多轮讨论，管理 Idea Pool 和创意路径历史 | STAGE 1 概念讨论 | `discuss` | Story DNA + 创意路径历史 |
| **Worldbuilder** 世界观师 | 基于 Story DNA 构建世界框架——时代、地理、战力/魔法体系、势力格局、科技背景 | STAGE 2 世界观工坊 | `discuss` | world.json |
| **Character Designer** 角色设计师 | 创建完整角色档案（人格/能力/关系/声音签名），在成长工坊中与用户协同设计成长曲线 | STAGE 2 角色工坊 | `discuss` | characters.json（含成长曲线） |
| **Outliner** 大纲师 | 规划分卷分章细纲 + 全局伏笔映射表，支持分支模拟推演，在情节头脑风暴中与用户反复推敲 | STAGE 3 情节头脑风暴 | `discuss` | outline.json（含伏笔映射 + 剧情-成长绑定） |
| **Writer** 写手 | 逐幕写作，嵌入 SF_LOG 变化标记，遵循 Style Engine 约束，支持创新豁免申请 | STAGE 4 逐章写作 | `live` | chapter_draft.md |
| **Reviewer** 审稿 | 三层审查（Fact/Narrative/Style Guard）+ 连贯性评分 + 主持章节评审会 | STAGE 4 每幕/章完成后 | `live` / `review` | review_report.json |
| **Summary Archiver** 归档员 | 生成章摘要、更新 MemoryOS 各层记忆、更新进度追踪、归档写作日志 | STAGE 4 每章完成后 | `auto` | 摘要 + 记忆更新 + progress.json |

### 协作原则

- 每个 Agent 在与用户对话时，以"创作伙伴"而非"工具"的口吻——主动提问、提供备选、解释原因、接受反驳
- Agent 之间的状态共享通过 Conductor 和 StoryOS 的结构化数据完成，而非 Agent 之间直接对话
- 用户在任意阶段都可以回溯到之前的 Agent 重新讨论——Conductor 执行回退影响传播分析（2.2 节），提醒受影响的章节和资产；影响范围 ≥ 10 章时自动建议使用叙事重构模式（6.8 节）

```yaml

project:
  name: "代码天才重生记"

  style_sample: "samples/"        # 可选：参考文本路径

  world_char_stage:   discuss     # discuss / review / approve
  plot_stage:         discuss     # discuss / review / approve
  writing_stage:      live        # live / approve / auto
  review_stage:       review      # review / approve / auto
  chapter_error_mode: retry3      # retry3 / stop / skip
  coherence_threshold: 80         # 低于此分触发重写

# v1.3 新增：回退与重构
rollback:
  auto_suggest_restructure: true  # 影响≥10章时自动建议叙事重构模式
  restructure_threshold: 10       # 触发叙事重构模式的影响章数阈值

# v1.3 新增：疲劳感知
fatigue_awareness:
  enabled: true
  rapid_skip_threshold: 10        # 连续快速通过N个Scene后提醒
  long_session_minutes: 90        # 单次会话超过此时长后提醒
  late_night_reminder: true       # 深夜写作提醒

memory:
  hot_window_chapters: 5
  cold_chunk_size: 500

  inherit: cool_novel
  overrides:
    min_beat_per_chapter: 3
    hook_required: true

      - "禁止出现真实公司名称"

reader_metrics:

  overrides:                      # 仅在 use_genre_defaults: true 时生效
    addiction_critical: 50        # 覆盖爽文默认的追更欲严重阈值
    # frustration_high: 60        # 如需自定义，取消注释并修改
    # fatigue_moderate: 55
```

## 二十一、产品价值总结

StoryForge 1.3 解决长篇小说创作的五个根本难题，**用户全程参与、全程掌控**：

1. **创意枯竭：** 套路库 + 四种变异操作 + 五个矛盾模板 + WhatIf 树 + 体裁融合器 + Global Trope Pool 市场数据。从 INIT 阶段开始，系统就在帮作者了解市场、发现蓝海方向。

2. **叙事失控：** 七类叙事资产显式追踪，跨资产关联自动级联（含冲突检测与隔离），全局伏笔映射表让每个伏笔都有生命周期。分支模拟引擎标注了确定性计算和 LLM 推理的边界——作者知道自己依据什么在做决策。叙事重构模式让后期的大胆偏离不再被系统复杂度压垮。

3. **一致性崩塌：** 角色四层档案 + 成长曲线绑定剧情里程碑 + 信念变化严格门槛 + Fact Guard 六项硬检 + 语义完整性预检 + 连贯性评分。确定性检查做阻断，概率性检查做建议——系统既不打盹也不过激。

4. **质量盲飞：** 七个读者体验指标 + 体裁差异化阈值 + 五维度连贯性评分 + 智能预警 + 章节评审会。不同体裁有不同的"健康指标"，爽文的追更欲阈值和严肃文学不一样。

5. **风格复用与创新平衡：** 三层风格控制 + 风格提炼 + 风格沙盒预览 + 创新豁免 + 跨项目风格模板。写完一部，风格资产自动带到下一部。

**v1.3 核心改进（相对于 v1.2）：**

| 改进项 | 变更类型 | 触发来源 |
|---|---|---|
| INIT 阶段完整规格 | 新增 | 审查：INIT 阶段几乎是空白 |
| Conductor 主控编排器详设（阶段状态机、回退影响传播、信号仲裁、熔断降级） | 新增 | 审查：Conductor 只有定义没有设计 |
| 跨作品资产复用（Global Trope Pool、Global Idea Pool、风格/体裁模板共享） | 新增 | 审查：跨作品资产复用几乎没有 |
| 叙事重构模式（批量处理全局创意偏离） | 新增 | 审查：结构性建议——系统越精确偏离成本越高 |
| 灵感路由器多分类冲突处理（优先级规则、多标签、用户纠错、置信度） | 增强 | 审查：分类边界没有冲突处理 |
| StoryOS 级联冲突检测与隔离（循环依赖检测、状态冲突检测、互斥检测） | 增强 | 审查：级联更新缺乏错误隔离 |
| ReaderOS 阈值基准说明 + 体裁差异化阈值 | 增强 | 审查：阈值来源从未说明、不同体裁应不同 |
| 分支模拟引擎 LLM/确定性边界标注 + 置信度提示 | 增强 | 审查：LLM 推演被写成确定性结果 |
| SF_LOG 语义完整性预检（高重要性事件的 LLM 辅助扫描，建议性质不阻断） | 增强 | 审查：SF_LOG 可靠性是最大系统性风险 |
| 协作疲劳感知（连续跳过检测、长会话提醒、深夜提醒） | 增强 | 审查：live 模式协作疲劳风险被低估 |
| Fact Guard 第 6 项硬检（语义预检结果复核——列出但不阻断） | 增强 | 上述 SF_LOG 增强的配套变更 |
| 阶段门控配置扩展（回退/重构阈值、疲劳感知开关、ReaderOS 阈值覆盖） | 增强 | 上述多项增强的配置入口 |
