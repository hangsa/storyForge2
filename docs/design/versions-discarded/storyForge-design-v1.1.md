# StoryForge 1.1 — 产品设计文档

> v1.1 变更：融合 design-origin 的完整创作阶段、Agent 分工体系、专家配置、修订流水线、进度追踪与 design-v1.0 的三能力链架构、创意引擎、叙事资产管理、变化标记机制、读者体验模型。

---

## 核心理念

StoryForge 是一个创意叙事操作系统，帮助作者创作长篇小说（以网文为主要场景）。系统融合了三类参考项目的最佳实践，围绕三个核心目标设计：

| 核心目标 | 能力链 | 负责系统 |
|---|---|---|
| 构思足够发散、足够创新 | Creative Chain | CreativeOS + Creative Director |
| 剧情足够反转、足够抓人 | Narrative Chain | StoryOS + ReaderOS + Scene Engine + Style Engine |
| 角色/剧情高度一致 | Consistency Chain | MemoryOS + State Machines |

**设计哲学：** 用规则与数据控制作品的骨架（一致性、节奏、读者体验），用 AI 填充血肉（创意发散、文字写作）。AI 负责创作，系统负责把关。

**核心差异化：** 相较于前代 AI 写作工具，StoryForge 独有的能力包括——写法公式自动提炼、五层记忆系统（含语义向量检索）、连贯性守卫独立 Agent、自动修订流水线、叙事资产显式追踪网络。

## 二、创作全流程状态机

### 2.1 六阶段创作流水线

```
INIT（项目初始化）
  │
  ▼
STAGE 1: CONCEPT_GEN（概念生成）
  输入：作者一句话意图 + 可选参考文本
  输出：concept.json（书名、体裁、前提、基调、目标读者、总章数、主题）
  │
  ▼ [人工审核门 1]
STAGE 2: WORLDBUILDING（世界观构建）
  输入：concept.json
  输出：world.json（时代设定、地理、战力体系、势力、科技背景）
  │
  ▼
STAGE 3: CHARACTER_DESIGN（角色设计）
  输入：concept.json + world.json
  输出：characters.json（主角档案 + 配角列表，含人格、能力、关系、声音签名）
  │
  ▼ [人工审核门 2]
STAGE 4: OUTLINE_PLANNING（大纲规划）
  输入：concept.json + world.json + characters.json
  输出：outline.json（分卷分章细纲 + 全局伏笔映射表）
  │
  ▼ [人工审核门 3]
STAGE 5: CHAPTER_LOOP（逐章写作循环）
  每章循环:
    ├── Writer 规划本章 Scene（3-6 幕）
    ├── Writer 逐幕写作 + 嵌入变化标记
    ├── Reviewer 三层审查（事实/叙事/风格）
    │     ├── 通过 → 继续
    │     └── 不通过 → 重写（最多 3 次）→ 熔断降级
    ├── StoryOS Agent 扫描标记 → 更新叙事资产 → 级联传播
    ├── MemoryOS 更新各层记忆
    ├── ReaderOS 重新计算读者指标
    └── Summary Archiver 摘要归档
  │
  ▼
STAGE 6: MANUSCRIPT_EXPORT（稿件导出）
  输出：多格式成品（Markdown / PDF / EPUB）
```

### 2.2 阶段门控配置

作者可配置每个阶段的审核模式：

| 阶段 | 可选模式 | 说明 |
|---|---|---|
| 概念审核 | `prompt` / `auto` / `skip` | prompt=需人工确认才继续 |
| 大纲审核 | `prompt` / `auto` / `skip` | auto=自动通过但保留回滚 |
| 章节写作 | `auto` | 自动执行 |
| 连贯性检查 | `retry3` / `stop` / `skip` | retry3=最多重写3次 |
| 连贯性阈值 | 0-100 | 低于此分数自动触发重写（默认80） |

## 三、Style Engine — 风格引擎

Style Engine 是 StoryForge 最核心的差异化组件之一。它不是简单的 Prompt 模板，而是将"如何写"系统化为可插拔的三层规则体系。

### 3.1 三层风格控制

**L1 体裁模板（Genre Templates）：**

预定义的流派写作规则，以 YAML 外置，体裁可插拔：

```
爽文     → 短句快节奏、每章≥3爽点、结尾必留钩子
严肃文学  → 长句铺陈、心理描写深度、节奏舒缓
悬疑推理  → 线索密度控制、信息不对称维持、反转节奏
科幻      → 设定自洽性、专业词汇阈值、科技感描写
奇幻      → 世界观一致性、魔法体系逻辑、种族设定
```

**L2 写作公式（Writing Formulas）：**

句子级别的定量约束：
- 句式模板：短句冲击 / 长句铺垫
- 情绪节奏模板：压抑→爆发→收获
- 对白节奏公式：对白/叙述比例
- 爽点密度公式：每章爽点数量与分布
- 环境描写公式：描写/行动比例

**L3 禁忌约束（Constraint Layer）：**

- 全局禁忌：禁止元引用（章节编号）、禁止真实公司名称、禁止虐主超限
- 角色禁忌：每个角色配置不可触犯的行为底线
- 体裁禁忌：特定体裁的禁止模式（如爽文禁止连续3段纯描写）

### 3.2 风格提炼（Style Extractor）

作者上传参考文本（已有章节、喜欢的作家作品），系统自动提取风格特征，生成可复用的 YAML 规则文件：

- **句式分析：** 平均句长、短句占比、感叹句密度
- **节奏分析：** 段落长度分布、对白/叙述比例
- **词汇分析：** 专业词密度、口语化程度
- **结构分析：** 每章爽点位置分布、高潮位置
- **提取结果：** 自动生成的 YAML 规则文件，可直接用于后续写作

## 四、CreativeOS — 创意引擎

### 4.1 设计目标

帮助作者从一个模糊的想法出发，系统化地发散出高新颖度、有市场潜力的故事设定。不替代作者创意，而是提供结构化创意工具。

### 4.2 七大创意模块

```
CreativeOS
├── Idea Pool              灵感种子库
├── Trope Pool             套路模式库（含市场饱和度）
├── Mutation Engine        套路变异器（4种变异操作）
├── Contradiction Engine   矛盾设定生成器（5个矛盾模板）
├── WhatIf Engine          连续发散器（递归树生成）
├── Genre Fusion Engine    体裁融合器（结构层面嫁接）
└── Novelty Evaluator      新颖度评估器（4维度打分）
```

### 4.3 套路知识库（Trope Pool）

系统化的网文套路知识库。每个套路记录：
- **核心机制：** 该套路怎么运作
- **市场饱和度（market_saturation）：** 0-1 之间的值，由人工标注 + 季度数据更新。低于 0.3 为"蓝海"，高于 0.8 为"红海"
- **子变体：** 该套路的常见变化形式
- **融合潜力：** 可跨界融合的其他套路
- **反模式：** 已烂大街的变体方向，建议避开

### 4.4 套路变异器（Mutation Engine）

给定一个基础套路，自动生成新颖变体。四种操作：

| 操作 | 说明 | 示例 |
|---|---|---|
| **逆转（Inversion）** | 反转核心预设 | 系统文 → "系统其实是骗局，在污染宿主意识" |
| **融合（Fusion）** | 跨体裁嫁接 | 重生 + 刑侦 = "凶手也重生了，且知道主角重生" |
| **升维（Escalation）** | 矛盾升级 | 主角无敌 → "无敌但无法伤害任何人" |
| **打破（Subversion）** | 颠覆读者预期 | 预设的反派 → 转变为守护者 |

每次变异输出：核心设定、由此产生的核心矛盾、新奇钩子、设定自洽性检查。

### 4.5 矛盾设定生成器（Contradiction Engine）

强设定的本质是内置矛盾。五个矛盾模板：

| 矛盾模板 | 含义 | 示例 |
|---|---|---|
| **能力 × 限制** | 强大能力有苛刻限制 | 能预知他人死亡，但无法预知自己的 |
| **永恒 × 消逝** | 永恒的事物会消逝 | 不死之身，但记忆每百年清零 |
| **身份 × 秘密** | 双重身份矛盾 | 表面是医生，实为器官贩卖网络的受害者 |
| **目标 × 代价** | 达成目标的代价是目标本身 | 追求力量为了保护，但力量在吞噬人性 |
| **力量即弱点** | 最强的力量恰是最大弱点 | 预知能力越强，越清楚一切不可改变 |

### 4.6 连续发散器（WhatIf Engine）

从一个核心前提出发，递归地进行"如果...会怎样？"的发散，生成创意树。默认深度3层、广度4个分支，最多可产生 84 个衍生节点。每个子节点必须是父节点的直接逻辑推论，保证发散逻辑性。

### 4.7 体裁融合器（Genre Fusion Engine）

在结构特征层面融合两个体裁，而非简单叠加标签：
- 维护每个体裁的**核心机制**、**读者承诺**、**兼容体裁**、**冲突体裁**
- 当两个体裁存在天然冲突时（如"系统文×刑侦"——系统给答案会破坏悬疑），设计化解方案
- 体裁距离越远，融合后新颖度加成越大（+5 / +15 / +30）

### 4.8 新颖度评估器（Novelty Evaluator）

从四个维度评估创意市场新颖度（0-100分）：

| 维度 | 权重 | 核心问题 |
|---|---|---|
| **市场饱和度** | 30% | 这个套路在市场上有多泛滥？ |
| **套路相似度** | 25% | 与已知套路有多相似？ |
| **矛盾深度** | 25% | 设定是否包含强内置矛盾？ |
| **讨论潜力** | 20% | 读者会不会为这个设定争论？ |

判定：≥80 高新颖度可推进 | ≥60 中等建议强化 | ≥40 偏低需显著变异 | <40 建议更换核心创意

**爆款潜力：** 新颖度(40%) + 反转密度(30%) + 讨论潜力(30%)

### 4.9 产出物：故事基因（Story DNA）

CreativeOS 最终产出结构化的"故事基因"文档：
- **核心矛盾：** 故事的根本张力
- **体裁融合方案：** 主体裁 + 融合体裁 + 嫁接点
- **选定的变异套路：** 从套路库选取并变异后的组合
- **关键创意节点：** WhatIf 发散的最优质分支
- **读者钩子设计：** 主谜团、主承诺、情感核心
- **创意约束条件：** 保障设定自洽的规则边界

## 五、StoryOS — 叙事资产管理

### 5.1 设计目标

长篇写作最大的挑战不是写不出来，而是忘了前期埋了什么。StoryOS 将叙事中的"无形资产"——冲突、承诺、谜团、反转、揭晓、目标、读者期待——显式登记、追踪、管理，确保每一个伏笔被回收，每一条剧情线有始有终。

### 5.2 七类叙事资产

| 资产类型 | 核心问题 | 关键属性 |
|---|---|---|
| **冲突（Conflict）** | 谁和谁在斗？斗到什么程度了？ | 类型（复仇/爱情/权力/生存/意识形态）、强度、预期解决章节 |
| **承诺（Promise）** | 角色对谁许下了什么？什么时候兑现？ | 说话人、听众、截止章节、兑现/打破/延期 |
| **谜团（Mystery）** | 什么悬念吊着读者？线索够了吗？ | 问题、已给线索、计划揭示章节、影响级别 |
| **反转（Twist）** | 什么颠覆在铺垫？读者预期的什么？ | 类型（身份/背叛/世界真相/能力来源/关系）、铺垫章节列表 |
| **揭晓（Reveal）** | 什么秘密要揭示？谁知道/不知道？ | 秘密内容、谁已知、读者是否已知、揭示方式 |
| **目标（Goal）** | 角色要达成什么？进展如何？ | 类型（短期/弧线/系列）、进度、阻碍/推动因素 |
| **读者期待（Expectation）** | 读者现在在期待什么？ | 期待内容、满足/颠覆、是否逾期 |

### 5.3 叙事资产关联网络

资产之间存在天然因果关系，系统自动追踪关联：

```
谜团被揭示 → 反转被触发 → 读者期待被满足
冲突升级   → 目标被阻碍
承诺兑现   → 期待被满足
揭示真相   → 冲突进一步升级
```

当一处发生变化时，系统自动推导并级联传播关联变化。

### 5.4 全局伏笔映射表

从大纲规划阶段就建立伏笔的完整生命周期追踪：

```
伏笔"超脑真相" → 埋设：第1章 → 计划揭示：第48章
伏笔"暗物质危机" → 埋设：第15章 → 计划揭示：第50章
伏笔"师父真实身份" → 埋设：第10章 → 计划揭示：第58章
```

每个伏笔追踪其状态（已埋设/铺垫中/可揭示/已揭示）、关联角色、影响级别。

### 5.5 全书张力曲线

实时计算全书张力值（0-100），基于活跃叙事资产的综合状态。自动预警：

- 连续3章张力<30 → "节奏平缓，需引入新冲突或推进悬念"
- 承诺即将超期（≤5章）→ "该兑现了，或给出延期理由"
- 读者期待逾期未兑现 → "读者等太久了，有弃书风险"
- 活跃冲突不足3个 → "戏剧性不足，建议衍生新冲突"
- 谜团超过10章未推进 → "读者可能已遗忘此悬念"

## 六、"变化标记"机制（SF_LOG）

### 6.1 设计目标

传统 AI 写作的核心问题：AI 在文字中隐含了角色关系变化、知识获取、冲突升级，但这些变化散落在文字中，难以被系统识别和追踪。StoryForge 的做法是——写作时 AI 在文本中嵌入"变化标记"（Markdown 注释语法，读者不可见），系统自动扫描提取，确定性更新，零 LLM 调用，不留遗漏。

### 6.2 变化标记类型

写作时，AI 在对应叙事位置嵌入以下标记：

```
角色关系改变    → 好感到冷淡、信任到裂痕、反目到和解
角色情绪变化    → 压抑愤怒、高度警觉、情绪爆发
角色获取知识    → 秘密、线索、真相
冲突升级/新增   → 冲突状态变化
谜团线索揭示    → 每条新线索
反转被触发      → 反转兑现时刻
读者期待被满足  → 完全满足/部分满足/颠覆
角色目标里程碑  → 阶段性进展
叙事资产创建    → 任何新的冲突/承诺/谜团等
```

### 6.3 工作流程

```
Writer 预声明本幕将产生的叙事变化
         +
Writer 写作文本并嵌入变化标记
         │
         ▼
StoryOS Agent 自动扫描标记（纯规则，零LLM）
  ├── 对比预声明与实际标记
  ├── 标记匹配 → 验证通过 → 应用变更
  ├── 预声明但无标记 → 记录遗漏
  └── 有标记但未预声明 → 接受（Writer主动新增）
         │
         ▼
级联传播：关联资产自动联动更新
  Mystery→revealed ⇒ Reveal→revealed ⇒ Expectation→fulfilled
  Reveal→revealed  ⇒ Conflict→escalated
  Twist→revealed   ⇒ Expectation→ready_to_fulfill
```

## 七、Scene Engine — 写作执行系统

### 7.1 设计目标

将叙事资产的规划落实到每一幕（Scene）的写作中。每个 Scene 带有明确的叙事任务：揭示哪条线索？升级哪个冲突？给读者什么体验？

### 7.2 Scene 的叙事规格

每个 Scene 写作前明确三个层面：

**叙事层：** 本幕目标 / 本幕冲突 / 情感弧线 / 叙事角色（小爽点 / 大爽点 / 铺垫 / 过渡 / 悬念结尾）

**读者体验层：** 是否有反转？兑现了什么？结尾钩子是什么？预期读者反应？

**叙事资产管理：** 涉及哪些叙事资产变动？需要嵌入哪些变化标记？

### 7.3 写作规范约束

通用写作规范（可按体裁覆盖）：

- 每章至少 4000 字
- 每章结尾必须有钩子
- 禁止"吃书"：严格遵守摘要中的已写内容
- 禁止元引用：不可引用章节编号
- 爽文特化：主角冲突后 3 章内完成反制、金手指须符合设定上限

## 八、MemoryOS — 五层记忆系统

### 8.1 设计目标

将不同时效的信息用不同存储介质管理，精准召回，不污染上下文。检索优先级：L0 → L1 → L4 → L2 → L3。

| 层级 | 内容 | 容量 | 使用场景 |
|---|---|---|---|
| **L0 运行时** | 当前章的即时状态：位置、情绪、活跃冲突、即将反转 | ~500 tokens | 每幕必须知道，始终在上下文顶部 |
| **L1 热记忆** | 近5章全文 + 关键细节清单 | ~15K tokens | 近期剧情连贯性 |
| **L2 温记忆** | 全书章摘要树、角色关系图、时间线 | ~8K tokens | 大局导航 |
| **L3 冷记忆** | 全书语义检索库（向量+关键词混合） | 按需召回 | 找回几十章前的细节 |
| **L4 叙事记忆** | 与 StoryOS 同步的叙事资产摘要 | ~3K tokens | 活跃伏笔、冲突、承诺一览 |

### 8.2 语义检索（L3 冷记忆）

全书内容向量化索引，支持：
- **语义检索：** "主角第一次使用超脑的场景"
- **关键词回退：** BM25 精确匹配作为语义检索的补充
- **交叉引用查询：** 某个人物/道具/地点在全书中的所有出现
- **混合排序：** 向量语义 + BM25 关键词，RRF 融合排名

### 8.3 细节防遗忘机制

AI 长篇写作倾向"遗忘"几章前的小细节。系统每 5 章自动运行"关键细节重提取"：扫描最近 5 章，提取所有"可能被遗忘但重要"的细节——装备/物品/伤情、未完成对话/约定、未探索线索——生成清单放在写作上下文顶部。

## 九、ReaderOS — 读者体验模型

### 9.1 设计目标

量化每章的读者阅读体验，全部基于可计算指标，不依赖 AI 的"感受"。

### 9.2 七个读者状态指标

| 指标 | 衡量什么 | 计算逻辑 |
|---|---|---|
| **好奇心** | 读者多想知道接下来发生什么？ | 当前开放谜团数量 × 重要性权重 |
| **张力** | 读者多紧张？ | 活跃冲突 + 即将反转的综合状态 |
| **满足感** | 读者最近爽了几次？ | 近3章兑现的期待和承诺数量 |
| **挫败感** | 主角是不是太惨了？ | 近5章主角目标被阻断的次数 |
| **疲劳度** | 读者是不是太累了？ | 连续高强度内容的持续时间 |
| **追更欲** | 读者会不会点下一章？ | 好奇心+张力+满足感+结尾钩子 |
| **讨论潜力** | 读者会不会在评论区吵？ | 道德模糊度+预期打破+文本争议性 |

### 9.3 自动预警

- **追更欲 < 40：** 严重——需检查结尾钩子和近期爽点密度
- **挫败感 > 70：** 高度——主角太惨，需安排胜利或进展
- **疲劳度 > 60：** 中度——持续高能，建议安排轻松缓冲
- **好奇心低但追更欲尚可：** 读者在"追习惯"而非"追内容"，需引入新谜团
- **张力高但讨论度低：** "爽但无争议"型章节，可增加道德模糊元素

## 十、一致性保障系统

### 10.1 角色一致性

每个重要角色维护完整人物档案：

**核心人格层（不轻变）：** 信念、欲望（短期/中期/深层）、恐惧、价值观

**当前状态层（随剧情更新）：** 位置、身体、情绪、关系、目标、已知/未知秘密

**能力进化追踪：** 角色能力的阶段性升级（如"一阶·绝对理智 → 二阶·万物互联 → 三阶·量子领域"），每阶段的能力上限和代价

**角色声音签名：** 说话风格、思维模式、绝对禁忌（角色绝不会做的事）

**信念变化的严格门槛：** 核心价值观变化需要充分的叙事事件支撑（至少 2 个独立事件），且变化必须由白名单内的触发事件驱动（被背叛、不可逆失去、道德觉醒、渐进式认知改变、关系质变等），不可随意漂移。

### 10.2 剧情线一致性

```
创建 → 激活 → 升级 → 解决
              ↘ 放弃（作者主动决定）
```

自动断线检测：超过 10 章未推进→"可能断线"；核心剧情线（importance=critical）→ 5 章阈值。

### 10.3 三层审稿机制

每幕写完后自动进行三层审查：

**第一层·事实核查（Fact Guard，硬性阻断）：**

五项确定性检查：
1. 时间线连续性——角色不能同时出现在两个地方
2. 角色状态一致性——不能展示不应该知道的秘密
3. 世界规则一致性——能力不超过设定上限（或付出代价）
4. 叙事资产合规——已解决的冲突不能被直接重新激活
5. 变化标记完整性——预声明的变化标记是否全部写入

不通过 → 自动打回重写，附带具体修改建议，最多重试 3 次。3 次仍不通过 → 触发熔断：标记问题 + 生成兼容性说明 + 强制通过 + 通知作者介入。

**第二层·叙事建议（Narrative Guard，软性建议，不阻断）：**
- 节奏是否合适、情绪是否连贯
- 角色行为是否有足够动机
- 角色行为是否与当前状态不符（状态漂移检测）

**第三层·风格标签（Style Guard，仅记录）：**
- 是否符合流派风格要求
- 是否触犯角色或流派禁忌

### 10.4 连贯性评分

综合评估每章质量（0-100分），低于 80 分自动触发重写：

| 维度 | 权重 | 扣分规则 |
|---|---|---|
| 时间线连续性 | 25% | 每处断层 -5 分 |
| 人物状态一致性 | 25% | 每处矛盾 -10 分 |
| 伏笔完整性 | 20% | 每个遗漏/意外 -5 分 |
| 世界规则一致性 | 20% | 每处违规 -8 分 |
| 禁忌约束 | 10% | 每处违禁 -15 分 |

## 十一、多轮修订流水线

### 11.1 全书自动诊断

系统可对全书进行一键质量诊断，分五大类问题：

- 时间线断裂（角色死亡后复活、事件时序矛盾）
- 角色不一致（能力前后矛盾、关系无故变化）
- 节奏问题（爽点密度分布、对话/叙述比例失衡）
- 重复冗余（高频短语、相似场景重复）
- 伏笔追踪（逾期未收、未规划的新伏笔）

### 11.2 分级修复策略

```
P0 阻断级（必须修复）:
  - 角色死亡后复活
  - 金手指能力前后矛盾
  - 时间线逻辑断裂

P1 质量级（应该修复）:
  - 重复短语超阈值
  - 角色工具化（多章无独立场景）
  - 爽点分布不均（连续3章无爽点）
  - 逾期未收回的伏笔

P2 优化级（可选修复）:
  - 过渡章节节奏生硬
  - 对话/叙述比例失衡
  - 场景描写密度不足
```

每级问题有对应的自动修复策略：时间线自动插入标记、为工具化角色补写独立场景、批量替换高频重复词、必要时生成过渡章节。

## 十二、Agent 协作体系

### 12.1 七 Agent 分工

| Agent | 职责 | 工作阶段 | 产出物 |
|---|---|---|---|
| **Creative Director** 创意总监 | 驱动所有创意引擎，产出故事基因 | 概念生成 | Story DNA |
| **Worldbuilder** 世界观师 | 构建时代、地理、战力、势力体系 | 世界观构建 | world.json |
| **Character Designer** 角色设计师 | 创建完整人物档案（人格、能力、关系） | 角色设计 | characters.json |
| **Outliner** 大纲师 | 规划分卷分章细纲 + 全局伏笔映射 | 大纲规划 | outline.json |
| **Writer** 写手 | 按幕写作，嵌入变化标记，遵循风格约束 | 章节写作循环 | chapter_draft.md |
| **Reviewer** 审稿 | 三层审查（事实/叙事/风格）+ 连贯性评分 | 每幕写完后 | review_report.json |
| **Summary Archiver** 归档员 | 生成摘要、更新记忆、归档进度 | 每章写完后 | summary + 记忆更新 |

### 12.2 人工门控

作者不是旁观者，在以下节点保持控制权：
- 概念生成后 → 确认创意方向
- 大纲产出后 → 审核章节规划
- 系统熔断时 → 3次重写仍不通过，通知介入
- 随时 → 查看和手动修改任何叙事资产
- 进度仪表盘 → 每章连贯性评分、字数的可视化追踪

## 十三、项目数据实体

### 13.1 核心文档结构

```
Project（一个作品）
├── concept.json          概念设定（书名、体裁、前提、基调、目标读者）
├── world.json            世界观（时代、地理、战力体系、势力、科技背景）
├── characters.json       角色档案（主角+配角，含能力进化阶段）
├── outline.json          大纲（分卷分章细纲 + 全局伏笔映射表）
├── style_formula.yaml    写法公式（从参考文本提取或手动配置）
├── progress.json         进度追踪（章状态、字数、连贯性评分）
├── .storyforge_checkpoint.json  断点续写快照
└── chapters/             章节文件
    ├── 001_死亡回档.md
    └── ...
```

### 13.2 进度追踪仪表盘

实时追踪整本书的创作进度：
- 已完成章数 / 总章数
- 总字数 / 平均每章字数
- 平均连贯性评分
- 每章的重写次数
- 写作日志（时间线记录）
- LLM 使用成本追踪

## 十六、产品价值总结

StoryForge 解决长篇小说创作的五个根本难题：

1. **创意枯竭：** 不是凭空想象，而是从系统化的套路库、矛盾模板、变异操作中生成有据可依的创意，每次产出都有市场新颖度评估。

2. **叙事失控：** 埋下的伏笔、承诺、冲突、谜团被显式追踪，每一个都有状态、有关联、有截止时间。断线自动检测，逾期自动提醒。全局伏笔映射表让作者对所有伏笔的生命周期一目了然。

3. **一致性崩塌：** 角色行为受限于其人格档案和当前状态，信念不会随意漂移，能力不会偷偷膨胀。每幕写完后自动通过事实核查。连贯性评分让质量可量化。

4. **质量盲飞：** 作者不再凭感觉判断质量——读者体验七指标、连贯性评分、张力曲线、修订流水线让作品质量可视化、可追踪、可迭代提升。

5. **风格复用：** 写法公式可提炼、可存储、可复用。一个成功的风格可以在多部作品中复用，不再是一次性的 Prompt 工程。
