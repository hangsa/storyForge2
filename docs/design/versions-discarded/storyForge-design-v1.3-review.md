# StoryForge v1.3 设计审查报告

## 总体评价

v1.3 对 v1.2 审查报告中指出的问题做了全面回应。INIT 阶段、Conductor 详设、跨作品复用、叙事重构模式四项新增填补了 v1.2 的主要空白；七项增强（级联冲突检测、ReaderOS 体裁差异化、分支模拟边界标注、语义预检等）均切中要害。文档整体质量显著提升，v1.2 审查中的 10 个问题有 9 个得到实质性解决，1 个（SF_LOG 可靠性）得到了显著改善。

以下按维度分析剩余问题和新增问题。

## 二、v1.3 新增问题

### 2.1 硬编码阈值与体裁差异化阈值的冲突

**问题：** 2.3 节信号仲裁表中 P1 严重预警写的是"追更欲 < 40"，但 14.4 节体裁差异化阈值表中爽文的追更欲严重阈值是 50、严肃文学是 30。"40"这个硬编码值既不是爽文也不是严肃文学的标准，意味不明。

**建议：** 2.3 节的信号仲裁表改为引用体裁差异化阈值，如"追更欲 < 体裁对应严重阈值"，或直接删除具体数字，仅保留"追更欲严重偏低"的描述。

### 2.2 Global Trope Pool 的市场数据更新机制不可操作

**问题：** 7.3 节将市场饱和度更新来源列为"系统内置：基于公开网文数据的季度批量更新"和"用户反馈：用户在灵感面板中标记"。但 StoryForge 作为本地写作工具，不存在服务端做"季度批量爬取"。Global Trope Pool 的实际可操作更新方式只有用户手动标记一种。

**建议：** 将"季度市场数据爬取"改为更诚实的描述——如"随 StoryForge 版本更新附带最新的市场饱和度快照（由开发团队维护）"。或者明确 Global Trope Pool 的市场数据初始值来自内置快照，后续演进完全依赖用户反馈。

### 2.3 STAGE 5 诊断未体现重构日志的作用

**问题：** 6.8 节强调 `restructuring_log.json` "在 STAGE 5 全书诊断时被特别关注——重构区域是诊断的重点扫描区"。但第五节 STAGE 5 的描述中完全没有提及重构日志的存在，诊断维度和流程中也没有"重构区域重点扫描"的任何展开。

**建议：** 在 5.6 节（STAGE 5）中增加一段"重构区域专项诊断"——说明系统如何利用 `restructuring_log.json` 对重构影响的章节进行额外的一致性检查（如新旧设定残留检测）。

### 2.4 语义预检的累积开销未评估

**问题：** 11.4 节语义完整性预检对每 Scene 的 3 种高重要性事件运行 Tier 3 模型（约 500 tokens/scene）。100 章 × 平均 4 Scene = 400 Scene，总计约 200K tokens 的额外开销。虽然 Tier 3 成本低，但这个开销在文档中没有任何说明。

**建议：** 在 11.4 节末尾或成本相关部分增加一行开销估算："单 Scene 预检约 500 tokens（Tier 3），整书 100 章累计约 200K tokens（约 $0.02），占 STAGE 4 总 token 预算的 < 0.2%。"

### 2.5 "七大协作机制"变成八个

**问题：** 第六节标题为"七大协作机制详设"，但内容包含 6.1-6.8 共八个子节（新增了 6.8 叙事重构模式），标题与实际内容数量不一致。

**建议：** 第六节标题改为"协作机制详设"，去掉数字，避免未来继续增减时的维护负担。或改为"八大协作机制详设"。

### 2.6 Fact Guard "零 LLM 调用"的描述不够精确

**问题：** 15.3 节写"六项检查（全部为规则匹配，零 LLM 调用）"。但第 6 项检查的输入——语义完整性预检结果——来自 11.4 节的 Tier 3 LLM 扫描。虽然第 6 项检查本身的执行逻辑是确定性的（展示列表 + 要求确认），但它的上游数据依赖 LLM。对读者来说"零 LLM 调用"可能产生误导。

**建议：** 将描述改为"前五项检查为纯规则匹配，零 LLM 调用；第 6 项为预检结果的确定性复核（预检本身使用 Tier 3 LLM，但复核逻辑不调用 LLM）"。

### 2.7 断点续写缺少 Mid-Scene 崩溃恢复

**问题:** 2.5 节断点续写以 Scene 为粒度，每个 Scene 完成后写入 checkpoint。如果系统在 Scene 写作过程中崩溃（Writer 正在生成文本但尚未提交到 Fact Guard），该 Scene 的所有进展丢失，需要从 Scene 开头重写。

**建议：** 考虑增加 Mid-Scene 草稿缓存——Writer 在生成每幕文本后立即将草稿写入临时文件（非 checkpoint，仅为崩溃恢复用）。恢复时 Conductor 检测到未完成的 Scene 有草稿缓存，询问用户是从草稿继续还是重新开始。

### 2.8 用户自定义矛盾模板的跨项目共享路径不明确

**问题：** 9.2 节写"用户可基于内置模板创建自定义矛盾模板，自定义模板自动保存到用户级模板库（第七节），跨项目可用"。但第七节的资产共享范围表中，Contradiction Templates 的存储层级为"系统级"，与这里的"用户级模板库"不一致。实际存储位置（`~/storyforge/global/contradiction_templates.json`）是正确的，但文字描述有矛盾。

**建议：** 统一表述：Contradiction Templates 中内置 5 个为系统级（随版本更新），用户自定义的为用户级（存储在 `~/storyforge/global/contradiction_templates.json`，跨项目共享）。

## 三、结构性观察

### 3.1 文档自包含性有退步

v1.2 中 world.json、characters.json、outline.json 的完整 JSON 结构都在文档中直接展示。v1.3 在 5.3、5.4 节将这三处标记为"结构同 v1.2，略"。对于未读过 v1.2 的读者，这造成了理解断层。作为"完整独立内容"（文档第 3 行声明）的设计文档，应保持自包含。

### 3.2 配置项与设计描述的同步风险

v1.3 在第十八节新增了三组配置项（rollback、fatigue_awareness、reader_metrics），但文档中对应的设计描述并未反向引用这些配置项。例如 3.2 节疲劳感知的判定条件写的是硬编码值（"连续 ≥ 10 个 Scene"），而配置项中 `rapid_skip_threshold: 10` 是可调整的。这类默认值与配置项的对应关系散落在文档各处，未来修改时容易出现配置改了但文档描述没改的情况。

### 3.3 模块间耦合度仍需关注

v1.3 新增的叙事重构模式（6.8 节）依赖 Conductor（影响范围计算）、StoryOS（叙事资产批量更新）、Outliner（大纲重设计）、Reviewer（重构对齐检查）四个模块协同。这比 v1.2 的任何单功能跨模块依赖都更复杂。当前文档每个模块独立成章，没有集中描述一个"叙事重构请求"在四个模块间的完整数据流和时序。这在实现阶段可能成为集成难点。

---

## 四、总结

v1.3 是一次高质量的增量改进。v1.2 审查指出的 10 个问题全部得到处理（9 个彻底解决 + 1 个显著改善），新引入的问题均为小型不一致或措辞精度问题，不涉及设计层面的系统性缺陷。

**优先修复（影响设计正确性）：**
- 2.1 硬编码阈值与体裁差异化阈值冲突
- 2.7 断点续写缺少 Mid-Scene 崩溃恢复
- 2.3 STAGE 5 未体现重构日志作用

**建议修复（提升一致性和可读性）：**
- 2.5 "七大协作机制"数量不对
- 2.6 "零 LLM 调用"描述不精确
- 2.8 矛盾模板存储层级描述不一致
- 3.1 文档自包含性退步

**可后续迭代（非阻塞）：**
- 2.2 Global Trope Pool 数据更新机制
- 2.4 语义预检开销评估
- 3.2 配置项与描述同步
- 3.3 模块间耦合数据流
