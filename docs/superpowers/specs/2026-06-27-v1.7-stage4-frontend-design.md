# T3.11-T3.12 Stage4 Frontend Integration — Design Spec

> **For agentic workers:** This is the design spec. Use `superpowers:writing-plans` to produce the implementation plan from this spec.

**Goal:** Surface precheck results, creative exemption approval, narrative impact, and SF log suggestions inside Stage4Page without bloating the existing 3-column layout.

**Architecture:** 4 new feature-scoped hooks + 7 new components + 3 new shared modules (`types/stage4.ts`, `auth.ts`, `uiStrings.ts`) + 1 Stage4Page rewrite + 1 small backend GET endpoint. Matches existing codebase pattern (`useConductor`, `useCreativeCanvas`).

**Tech Stack:** React 18 + TypeScript + Vite + Vitest + React Testing Library. Material Symbols Outlined icons. Existing `GlassPanel` shared component.

---

## Background

StoryForge v1.7 backend now exposes four features that previously had no UI:

1. **Semantic precheck** (Plan 2) — `precheck_result` is included in `writeScene` response. Lists missing SF_LOG tags that the Tier 3 LLM thinks should be in the scene.
2. **Creative exemption** (Plan 3) — `ExemptionManager` + 5 HTTP endpoints. Writer submits exemption requests; user approves/rejects.
3. **User edit assist** (Plan 4) — `SFLogSuggestionEngine` + 2 HTTP endpoints. After user edits draft, analyze diff → propose SF_LOG tags.
4. **Narrative impact** (existing since v1.6) — `/conductor/analyze-impact` endpoint, currently only accessible via `/project/:id/impact` route.

Currently Stage4Page has a 3-column layout (left = draft canvas, right = scene nav + ReaderOS + formulas + SF log feed + Fact Guard). Adding 4 more panels inline would make the right column very long.

## Design Decisions (from brainstorming 2026-06-27)

| Decision | Choice | Rationale |
|---|---|---|
| Layout | Bottom drawer with 4 tabs | Keeps canvas tall, dedicated home for review work, doesn't bloat existing panels |
| Drawer visibility | Auto-open for action-required only (exemption) + toast for informational (precheck, suggestions) + badge when collapsed | Auto-opening for informational results interrupts mid-edit; toasts surface the info without yanking focus. Auto-open is reserved for decisions the user must make. |
| Exemption UX | Modal for full context | Approval needs more space than inline; full context (rule, draft excerpt, antipatterns) prevents poor decisions |
| SF log UX | Dual-section card list (Deleted Tags / Suggested Tags) | Clear separation between info-only deletions and actionable suggestions |
| Narrative impact | Embedded compact panel inside drawer | User gets summary without leaving Stage4; can click through for full report |
| Architecture | Feature-scoped hooks (B) | Matches existing `useConductor`/`useCreativeCanvas` pattern; single-responsibility; testable in isolation |
| String source | Centralized `uiStrings.ts` | Future i18n is a file swap, not a code search |
| Apply flow | Server-side PUT `/sf-logs` (existing endpoint) | Server is the only authority on SF_LOG insertion order; no client-side substring logic that can diverge |

### Drawer behavior

- **Peek state** (default when collapsed): 60px tall strip showing 4 tab labels with colored count badges (red for action-required like exemption-pending, blue for informational like precheck-suggestions).
- **Expanded state**: 360px tall, shows active tab content.
- **Auto-open (action-required only)**: when the exemption count goes from 0 → >0, the drawer auto-expands to the Exemption tab. The user must take action; leaving it unseen has a cost. For precheck and SF_LOG suggestions (informational), the drawer does **not** auto-open — see Toast notifications below.
- **Toast notifications**: when precheck or SF_LOG suggestions arrive, show a non-blocking toast in the corner (`uiStrings.toasts.precheckReady(n)` / `uiStrings.toasts.suggestionsReady(n)`) for `TOAST_AUTO_DISMISS_MS`. The user clicks the toast to open the drawer on that tab. Toasts stack vertically (max 3 visible; older ones dismiss early).
- **User collapse**: clicking the drawer header (or pressing Escape on focused drawer) returns to peek.
- **Auto-expand on mount**: no. Always starts collapsed.

### Visual language

- Drawer header: `GlassPanel` style with bottom-rounded corners (mirrors the top panels).
- Active tab: bold text + accent color underline.
- Tab badge: small pill with number, colored by urgency (red = needs action, blue = informational, gray = empty).
- Drawer slides up/down with a 200ms CSS transition.

### `SFLogSuggestionsTab.tsx`

Props:
```ts
interface SFLogSuggestionsTabProps {
  data: SFLogDiffReport | null;
  loading: boolean;
  error: string | null;
  onApply: (suggestions: SFLogSuggestion[]) => Promise<void>;
  onDismiss: () => void;
}
```

Renders: two sections.
- **`uiStrings.sfLog.deletedSection`** (info-only, gray): list of deleted_logs. Each shows raw_text. No actions.
- **`uiStrings.sfLog.suggestedSection`** (actionable, accent color): list of suggestions. Each card shows event_type + reason + full suggested_tag (in `<code>`) + per-card `uiStrings.sfLog.insertOne` button. Top of section: `uiStrings.sfLog.insertAll` button. Both buttons call `onApply(suggestions)` which hits the server's `apply_sf_log_suggestions` endpoint (PUT `/sf-logs`). Server returns `{ updated_text }`; client then calls `updateSceneDraft` with the returned text and clears the report. Insertion order is determined by `location_hint` (top-to-bottom in text) — the server handles sorting.

If `data === null`: shows `uiStrings.sfLog.empty`. After dismiss, data clears. While `apply()` is in flight, both Insert buttons are disabled and show a spinner. The server is idempotent: re-applying the same suggestions is a no-op (verified by a test in the test plan below).

### `useStage4Exemptions.ts`

```ts
interface UseStage4ExemptionsReturn {
  items: ExemptionRequest[];
  loading: boolean;
  error: string | null;
  approve: (id: string, approvedBy: string) => Promise<void>;
  reject: (id: string, reason: string) => Promise<void>;
  refresh: () => Promise<void>;
  getAntipatterns: (id: string) => Promise<ExemptionAntipattern[]>;
}
```

`refresh()` calls `api.listExemptions(projectId, "pending")` (new API method, see Backend section).
`approve(id, approvedBy)` calls `api.approveExemption(projectId, id, approvedBy)` then `refresh()`. `approvedBy` is a per-app identifier — v1 imports `APPROVER_DEFAULT = "user_default"` from `frontend/src/auth.ts` (no auth system yet; UI shows the constant as the approver name). When auth is added later, this becomes the current user's ID with the import being the only change site.
`reject(id, reason)` calls `api.rejectExemption(projectId, id, reason)` then `refresh()`.
`getAntipatterns(id)` calls `api.getExemptionAntipatterns(projectId, id)` (server endpoint `GET /{exemption_id}/antipatterns` already exists). Returns the antipattern list. Errors fall through to a returned empty array — antipatterns are advisory, not blocking.

### `useStage4Suggestions.ts`

```ts
interface SFLogSuggestion {
  type: "missing" | "modified";
  severity: "warning" | "suggestion";
  event_type: string;
  suggested_tag: string;
  location_hint: string;
  reason: string;
}

interface SFLogDiffReport {
  original_text: string;
  modified_text: string;
  deleted_logs: Array<{ raw_text: string; type: string; id: string }>;
  suggestions: SFLogSuggestion[];
  tokens_used: number;
}

interface UseStage4SuggestionsReturn {
  report: SFLogDiffReport | null;
  loading: boolean;
  error: string | null;
  analyze: (original: string, modified: string, charNames: string[]) => Promise<void>;
  apply: (suggestions: SFLogSuggestion[], currentText: string) => Promise<{ updated_text: string }>;
  clear: () => void;
}
```

`analyze()` calls `api.suggestSFLogChanges(projectId, sceneId, original, modified)`.
`apply()` calls `api.applySFLogSuggestions(projectId, sceneId, currentText, suggestions)` (server's PUT `/sf-logs` endpoint). Returns the server's `{ updated_text }` so the caller can persist it. The server is the only source of truth for insertion order and formatting — no client-side substring logic. Stage4Page then calls `api.updateSceneDraft(...)` with the returned text and `clear()`s the report. `apply()` shares the same `loading` flag as `analyze()`.

```python
def list_exemptions(project_id: str, status: str = "pending") -> list[dict]:
```

## Stage4Page rewrite

The existing 640-line `Stage4Page.tsx` is rewritten to:

1. Keep all existing imports + state + handlers (writeScene, forcePass, skip, loadDraft, draftEdit, advance, etc.).
2. Add 4 new hooks: `useStage4Precheck`, `useStage4Impact`, `useStage4Exemptions`, `useStage4Suggestions`.
3. Add drawer state: `activeTab` (`DrawerTab | null`).
4. Add `useToast()` for non-blocking notifications (already exists in the codebase per the BranchSimulation page; reuse it).
5. After `handleWrite` success: call `precheck.setData(response.precheck_result)` and `toast.show(uiStrings.toasts.precheckReady(n))` (do not change `activeTab`).
6. After `handleDraftEditSave` success: if old ≠ new, call `suggestions.analyze(old, new, characterNames)`. The hook is fire-and-forget — no blocking UI. On completion, `toast.show(uiStrings.toasts.suggestionsReady(n))` runs from inside the hook (the hook holds a `toast` ref). Use an `AbortController` to cancel the in-flight `analyze()` if the user navigates away or the scene changes.
7. Render `<Stage4Drawer>` after the closing `</div>` of the main grid.
8. Wire effects for `activeTab`:
   - `exemptions.items.length` goes 0 → >0: `setActiveTab("exemption")` (auto-open, action-required).
   - `precheck.data` populates / `suggestions.report` populates: do not change `activeTab`; show toast instead.

### Data lifecycle rules

Per-scene data (precheck, suggestions) is bound to a specific `(project, chapter, scene)`. On scene navigation:

| Event | precheck | suggestions | exemptions |
|---|---|---|---|
| Mount Stage4Page | cleared | cleared | refreshed |
| Advance to next scene (handleAdvance) | cleared | cleared | refreshed |
| Jump to scene N via scene nav | cleared | cleared | not refreshed (still project-scoped) |
| Navigate away from Stage4Page | cleared | cleared (cancels in-flight `analyze` via AbortController) | cleared |
| Refresh button on Exemption tab | not touched | not touched | refreshed |

Exemptions are project-scoped (writer submits per-scene but the queue is project-wide), so they don't clear on scene change. Per-scene data must clear so the badges don't lie about stale results.

```tsx
            await api.updateSceneDraft(projectId, chapterNumber, sceneNumber, updated_text);
            suggestions.clear();
          }} onDismiss={suggestions.clear} />,
        }}
      </Stage4Drawer>

      {approvalItem && (
        <ExemptionApprovalModal
          item={approvalItem}
          onApprove={async () => { await exemptions.approve(approvalItem.id, APPROVER_DEFAULT); setApprovalItem(null); }}
          onReject={async (reason) => { await exemptions.reject(approvalItem.id, reason); setApprovalItem(null); }}
          onClose={() => setApprovalItem(null)}
        />
      )}
    </>
  );
}
```

## Error handling

- Every hook returns `{ data/report/items, loading, error }`. No thrown promises across the UI boundary.
- All API calls wrapped in try/catch. Failures surface as `error` string in the hook state, which the tab renders as a small banner ("加载失败：LLM 不可用") rather than crashing the page.
- Precheck `skipped_reason` renders as info note (not error): "预检已跳过：no router".
- Drawer never blocks the page. Tabs that fail to load show their error inline.
- Modal (ExemptionApprovalModal) is dismissible via Escape or X button — never trapped.

## Risk Notes

1. **Drawer auto-open is restricted to action-required tabs only**: only the Exemption tab auto-opens (exemption items are decisions the user must make). Precheck and SF_LOG suggestions are informational and arrive as toasts, so the drawer never yanks the user away from a draft they're editing. Escape + click-to-collapse + collapsing returns to peek (not full hide).
2. **Backend `list_exemptions` is a small new endpoint** (~15 lines) but is on the critical path for the ExemptionTab. If backend changes break it, the tab fails silently (empty list).
3. **`useStage4Suggestions.apply()` calls server PUT `/sf-logs`**: the server is the only authority on SF_LOG insertion. Client sends `{ text, suggestions }`, receives `{ updated_text }`, and persists via `updateSceneDraft`. This removes the divergence risk that existed when the client did local substring insertion.
4. **Modal Escape-key handling**: must not conflict with browser-level Escape (e.g., fullscreen exit). Use `keydown` listener scoped to the modal element. If the user has unsaved reason-text edits, prompt "放弃未保存的拒绝原因？" before closing.
5. **Drawer + existing circuit breaker modal**: both are fixed-position overlays. They must not appear simultaneously. Mitigation: circuit breaker modal renders above drawer (z-index 60 vs drawer's 50). If a circuit breaker is firing while the drawer is expanded, drawer collapses to peek; auto-open is suppressed until the circuit breaker resolves.
6. **In-flight `suggestions.analyze()` on scene change**: cancelled via AbortController. If the abort fails (e.g., server already responded), the response is dropped because the hook's `data` ref is checked against the current scene id before `setData` runs.
7. **Exemption list refresh after approve/reject race**: if the user clicks Approve then Reject on two different items quickly, the second refresh wins. Acceptable for v1 (both actions are user-initiated and rare to chain). Future work: optimistic update with rollback on refresh failure.

## Accessibility requirements

- **Tab semantics**: `Stage4Drawer` peek strip is `role="tablist"`. Each tab is `role="tab"` with `aria-selected` and `aria-controls` pointing to the panel id. The active panel is `role="tabpanel"` with `aria-labelledby` pointing to its tab.
- **Roving tabindex**: only the active tab has `tabindex={0}`; others have `tabindex={-1}`. Arrow Left/Right moves focus and selection. Home/End jump to first/last tab. Tab key from inside the panel moves focus out of the drawer.
- **Focus management**: opening the drawer moves focus to the active tab (or first tab if none). Collapsing returns focus to the element that opened the drawer.
- **Live regions**: peek strip has `aria-live="polite"` so badge count changes are announced ("3 new precheck suggestions"). Toasts use `role="status"` (polite) for precheck/suggestions. Exemption auto-open uses `aria-live="assertive"` so the user is informed even if focus is elsewhere.
- **Modal focus trap**: `ExemptionApprovalModal` traps focus within itself. First focusable element receives focus on open. Tab/Shift+Tab cycle within the modal. Escape closes (after confirm if reason-text is unsaved). Uses `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing to the rule description heading.
- **Loading states**: tabs in loading state have `aria-busy="true"` on the tabpanel. The page itself is never `aria-busy` — other interactions remain available.
- **Color contrast**: badge colors must pass WCAG AA (4.5:1 for text, 3:1 for graphics) against the `GlassPanel` background. Verify before merging.
- **Keyboard parity**: all drawer + tab + card + modal interactions are reachable via keyboard alone. Tested manually and via `userEvent` in RTL tests.
