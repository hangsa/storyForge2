## 1. 背景与目标

**问题：** 用户的灵感一闪而过（讨论中突然想到"如果 X 是 Y 的卧底会怎样"），当前没有任何捕捉机制，依赖用户手动记忆。

**目标：** 在所有阶段的 Agent 对话和用户交互中**持续运行**，自动捕捉、分类、存储灵感，避免遗忘。

**关键设计原则：**
- **零侵入**：路由器是后台进程，不阻塞主流程
- **去重**：相似文本（≥0.85 相似度）合并，避免 Idea Pool 膨胀
- **可纠错**：用户可重新分类/拆分/合并/标记无效

**触发实现：**

- **不**拦截 LLM 调用流（旁路，避免增加延迟）

## 5. 分类逻辑（Tier 3 + 规则）

```python
# backend/inspiration/classifier.py
CATEGORIES = ["设定灵感", "剧情想法", "角色灵感", "风格偏好", "写作灵感"]

@dataclass
class ClassificationResult:
    category: str                          # 主分类
    confidence: float                       # 0-1
    secondary_category: str | None         # 跨类标记
    secondary_confidence: float | None
    needs_review: bool                      # 0.5-0.8 区间
```

**分类规则（与设计文档一致）：**

| 置信度区间 | 处理 |
|---|---|
| ≥ 0.8 单一分类 | 直接写入，status=active |
| 0.5-0.8 | 写入，status=needs_review，UI 显示待确认标签 |
| 多分类差距 > 0.3 | 归类到最高置信度 |
| 多分类差距 ≤ 0.3 | 双标签跨类（如"跨类-设定/剧情"） |

**Prompt：** `backend/prompts/inspiration/classify.yaml`
- 输入：用户文本（≤500 字）
- 输出：5 类置信度 JSON 数组
- Tier 3 (Haiku)，~50 in + 30 out tokens

## 6. 去重策略

```python
# backend/inspiration/dedup.py
def should_merge(new_idea: Idea, existing: list[Idea], threshold: float = 0.85) -> Idea | None:
    """Compare new idea against existing; return existing to merge into, or None."""
```

**方法：**
1. 优先关键词重叠（确定性，cheap）：3 个以上关键词重合 → 候选
2. bge-m3 embedding 余弦相似度（已有 L3 基础设施）：≥ 0.85 → 合并

**合并策略：** 保留更长的 content，confidence 取 max，source_context 合并。

## 7. 数据模型

Idea 模型已存在于 `backend/models/creative_os.py`，**新增字段：**

```python
class Idea(BaseModel):
    # ... 已有字段 ...
    needs_review: bool = False           # 0.5-0.8 置信度标记
    cross_category: str | None = None    # "跨类-设定/剧情" 等
    marked_invalid: bool = False         # 用户标记噪音
    merged_into: str | None = None       # 被合并到哪条 idea 的 id
```

**用户纠错操作（PATCH 端点）：**
- `reclassify` → 修改 category / cross_category
- `split` → 拆分为多条（API 返回新 ids，原 idea 标记 merged_into）
- `merge` → 合并到另一条
- `mark_invalid` → 设置 marked_invalid=true
- `edit_content` → 修改 content 文本

## 8. API 设计

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/v1/projects/{id}/inspirations` | 手动触发捕捉（也作为内部 trigger 入口） |
| GET | `/api/v1/projects/{id}/inspirations` | 列表（支持 ?category=&source_stage=&status=） |
| PATCH | `/api/v1/projects/{id}/inspirations/{idea_id}` | 用户纠错（reclassify/split/merge/mark_invalid/edit） |
| DELETE | `/api/v1/projects/{id}/inspirations/{idea_id}` | 删除 |
| POST | `/api/v1/projects/{id}/inspirations/capture` | 全局手动捕捉（前端"捕捉灵感"按钮） |

**响应统一格式：** `{"error": False, "code": "OK", "detail": {...}}`

**错误处理：**
- Tier 3 不可用 → 返回 `{"skipped_reason": "no router"}`，idea 暂存为 `pending_classification`，下次启动时补分类
- 相似度合并导致 idea 数 > 100 → 提示用户"灵感池过大，请清理无效项"

## 11. 验收 / 测试

**单元测试（backend，≥12）：**
- classifier Tier 3 路径 + no-router 降级 + 多标签跨类
- dedup 关键词重叠 + embedding 阈值
- trigger_registry 8 阶段覆盖
- IdeaPool 已有测试（继承）+ needs_review / cross_category 字段
- 5 个 API 端点的 happy path + 错误处理

**前端测试（≥6）：**
- IdeaPoolPanel 渲染筛选 / 列表 / 待确认区
- 纠错操作触发 PATCH → 列表更新
- 全局"捕捉灵感"按钮触发 POST
- 设置页配置项读取与保存

**AC-8 端到端：** 模拟 10 轮 Agent 对话 → 断言 `len(captured_ideas) >= 3` + 人工验证 5 类分类准确率 ≥ 70%

**集成性能：** 10 轮对话触发的分类 + 去重总耗时 < 2 秒（Tier 3 × 10 = ~1s，embedding × 10 = ~0.5s，开销可接受）

## 12. 待对齐问题

1. **跨项目共享：** 设计文档说"v1.8 才做"，确认 v1.7 不做全局灵感池？倾向**确认不做**。
2. **触发粒度：** Agent 每次 call 都触发，还是只在 call 输出"包含新信息"时触发？倾向**每次 call**（简单一致）+ 后续去重解决膨胀。
3. **embedding 复用：** 用 L3 已有的 bge-m3 还是新建？倾向**复用 L3**（已有基础设施，零成本）。
4. **手动捕捉的文本来源：** 仅手动输入，还是允许"选中聊天消息 → 自动捕捉选中段"？倾向**两者都支持**，UX 更好。
5. **优先级与 Phase 4 顺序：** 这是 3 个里 greenfield 风险最大。是否考虑砍掉/降级为 v1.7.1 patch？
